# Step 04 自动保存策略收敛评审

- 日期: 2026-04-07
- Step: 04
- 当前能力等级: L3
- 评审范围: `NotesWorkspacePage.tsx` 自动保存调度链路、页面隐藏时强制落盘策略、workspace contract 挂载完整性

## 1. 评审结论

本轮增量已将自动保存的“是否调度、是否在 `pagehide` 强制刷盘、延迟时间常量”从页面组件中抽离到独立服务 `noteWorkspaceAutosave.ts`，并通过 Node contract 固化行为。页面层保留 React 生命周期与副作用挂载，服务层负责纯决策，边界清晰度明显提升。

本次收敛后，自动保存链路满足以下要求：

1. 仅在存在活动笔记、笔记未被删除、保存状态为 `dirty` 时调度自动保存。
2. 自动保存延迟统一由服务常量 `CANVAS_WORKSPACE_AUTOSAVE_DELAY_MS` 管理，避免页面层散落魔法值。
3. 页面隐藏事件与定时器共享同一决策来源，降低生命周期分支漂移风险。
4. `test:workspace:contracts` 已纳入 autosave contract，后续回归不再依赖人工记忆补跑。

## 2. 设计与实现评审

### 2.1 设计合理性

- 优点: 页面副作用与保存判定逻辑解耦，后续迁移到后台同步、本地草稿缓存或冲突解决时，可在服务层扩展策略，而不需要直接修改页面生命周期代码。
- 优点: 自动保存策略当前是纯函数，天然适合 TDD、静态推演和跨端复用。
- 约束: 页面仍然负责挂载 `setTimeout` 与 `pagehide` 监听，因此生命周期编排还未完全从 `NotesWorkspacePage.tsx` 收拢出去。

### 2.2 实现逻辑评审

- 服务输入只依赖 `activeNoteId`、`activeNoteDeletedAt`、`saveState`，依赖面足够收敛。
- 返回值采用 `shouldSchedule / shouldFlushOnPageHide / delayMs` 结构，可读性优于页面内联条件判断。
- `dirty` 且未删除的活动笔记是当前唯一允许自动保存的触发条件，符合“最小必要写入”原则。

### 2.3 架构质量评估

- 高内聚: 自动保存决策集中在单一服务文件中，达成。
- 低耦合: 页面对自动保存的理解降为“消费计划”，而不是“推导规则”，达成。
- 易扩展: 可以继续扩展为 `flushReason`、`syncPriority`、`localDraftFallback`，基本达成。
- 易维护: 行为入口固定，测试成本低，达成。

## 5. 当前缺口

Step 04 仍应维持 `L3`，原因如下：

1. 页面层仍保留 `flushDraft`、`useEffect` 定时器装配、`pagehide` 事件装配，生命周期编排尚未抽象为更高层的 runtime coordinator。
2. 对话框确认流、恢复笔记流、创建笔记流等页面级运行时逻辑仍然分散在 `NotesWorkspacePage.tsx`。
3. 自动保存仅完成“决策抽离”，尚未形成“运行时装配抽离”。

## 6. 下一步建议

建议继续围绕 `NotesWorkspacePage.tsx` 做下一轮 TDD 收敛，优先级如下：

1. 抽离 autosave runtime coordinator，将 `setTimeout/pagehide/flush` 生命周期绑定整体下沉。
2. 或抽离 dialog runtime coordinator，将 `pendingDialog -> confirm -> command execution` 闭环收敛为单一服务。
3. 完成上述任一方向后，再重新评估 Step 04 是否逼近 `L4`。
