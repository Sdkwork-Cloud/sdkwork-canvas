# 自动保存策略补充设计

- 日期: 2026-04-07
- 所属 Step: 04
- 主题: 页面自动保存策略边界收敛

## 1. 背景

在当前笔记工作区中，自动保存涉及三个不同层面的职责：

1. 业务决策: 何时允许自动保存。
2. 页面运行时: 何时挂载定时器、何时响应 `pagehide`。
3. Store 持久化: 实际调用 `persistActiveNote()` 落盘。

如果这三类职责同时堆叠在 `NotesWorkspacePage.tsx`，会导致页面承担“规则定义 + 生命周期调度 + 写入调用”三重责任，不利于后续扩展离线草稿、同步优先级和写入熔断策略。

## 2. 当前设计

本轮补充后，自动保存采用“计划驱动”的轻量分层：

### 2.1 策略层

文件: `packages/sdkwork-canvas-canvas/src/services/noteWorkspaceAutosave.ts`

职责:

- 根据活动笔记状态生成 autosave plan。
- 统一管理自动保存延迟常量。
- 为页面运行时提供纯函数输出，不直接依赖 DOM、Hook 或 Store。

输入:

- `activeNoteId`
- `activeNoteDeletedAt`
- `saveState`

输出:

- `shouldSchedule`
- `shouldFlushOnPageHide`
- `delayMs`

### 2.2 页面运行时层

文件: `packages/sdkwork-canvas-canvas/src/pages/NotesWorkspacePage.tsx`

职责:

- 消费 autosave plan。
- 挂载定时器与 `pagehide` 事件。
- 在触发时调用 `persistActiveNote()`。

约束:

- 页面不再定义“脏笔记判定逻辑”。
- 页面仅负责运行时绑定，不负责业务规则推导。

## 3. 架构价值

### 3.1 高内聚

自动保存策略集中在单一服务中，避免多个 Hook 分支各自维护条件。

### 3.2 低耦合

页面只依赖 plan，不依赖规则细节。策略层也不感知 React 生命周期实现。

### 3.3 易扩展

后续可以在不破坏页面生命周期代码的前提下扩展：

- 不同笔记类型差异化延迟。
- 本地草稿优先与远端刷盘降级策略。
- 网络异常时的写入退避和重试。
- 页面关闭前的强制 flush 优先级。

## 4. 接口与评估标准

| 维度 | 标准 | 当前状态 |
| --- | --- | --- |
| 输入收敛 | 策略输入必须仅包含业务状态，不包含 React/DOM 对象 | 达成 |
| 输出稳定 | 页面消费的 plan 结构必须语义明确且可测试 | 达成 |
| 生命周期一致性 | 定时保存与页面隐藏保存必须由同一 plan 驱动 | 达成 |
| 扩展能力 | 新增保存策略时不应直接修改多个页面 Hook 判断 | 基本达成 |
| 复用能力 | 策略函数需支持未来桌面端或移动端复用 | 达成 |

## 5. 行业对标视角

对标 Notion、Obsidian、Apple Notes 等领先笔记应用，自动保存能力不应只是“能保存”，而应满足以下先进标准：

1. 保存规则可解释，便于审计和问题定位。
2. 生命周期触发一致，避免不同入口行为不一致。
3. 后续支持离线、本地缓存、同步冲突时，可在策略层逐步演进，而不是重写页面。

当前实现已完成第一阶段基础收敛，但距离行业领先水平仍需继续补齐 runtime coordinator、本地草稿缓冲、异常退避与同步感知等能力。
