# SDKWork Canvas 分步实施计划索引

## 1. 文档定位

本目录是 `docs/架构/` 的执行层文档，负责把已经冻结的架构基线进一步细化为可以逐步落地的实施步骤、检查点、测试计划、结果验证和并行执行编排。

本目录不是替代：

- `docs/架构/README.md` 的主索引与冻结结论
- `docs/架构/01-10` 的现状事实、目标架构和实施路线

而是对上述文档做执行级拆分，形成“按 step 推进、每步可验收、每步可回滚、每步可复盘、可按车道并行”的闭环计划体系。

## 2. 适用范围

本计划适用于当前 `sdkwork-canvas` 仓库的以下真实结构：

- `sdkwork-canvas-pc-react/src`
- `sdkwork-canvas-pc-react/packages/*`
- `sdkwork-canvas-pc-react/scripts/*`
- `.github/workflows/*`
- `docs/架构/*`

本计划默认当前仓库是一个已经可本地运行、具备 Web/Desktop 双入口、但尚未完成本地优先、搜索、同步、版本恢复、可信发布链和高层自动化测试收敛的 React + Tauri 笔记应用。

### 4.1 Step 完成后的能力兑现口径

每个 Step 完成时，必须至少兑现下表中的能力切片；如果只完成局部实现、未形成能力闭环，不允许判定为 Step 完成：

| Step | 完成后必须兑现的能力切片 | 未达到则不得宣称完成 |
| --- | --- | --- |
| `00` | 执行门禁、证据口径、回滚口径、检查点命名统一冻结 | 后续 Step 仍可按各自口径推进 |
| `01` | 现状基线、差距矩阵、风险优先级、能力映射冻结 | 后续 Step 仍基于猜测推进 |
| `02` | 共享壳层、包落点、导出面、依赖方向冻结 | 新能力仍默认堆进 `canvas-canvas` |
| `03` | 会话安全迁移路径、配置治理分层、敏感项治理冻结 | 敏感会话仍长期依赖高风险持久层 |
| `04` | repository / orchestration / selector / page 边界稳定 | 本地层、搜索、同步接入仍需重写页面 |
| `05` | 编辑器保存状态机、失败恢复、切换一致性闭环 | 断网/崩溃/切换仍易丢稿 |
| `06` | 本地存储 schema、权威源、离线草稿和恢复闭环 | 本地层仍只是临时缓存 |
| `07` | 统一索引 schema、统一查询语义、统一检索入口闭环 | 搜索与命令面板仍然两套逻辑 |
| `08` | 同步队列、重试、冲突状态和恢复交互闭环 | 失败仍只能中断或静默覆盖 |
| `09` | 桌面 runtime 边界、可信发布链、dry-run 闭环 | 只能本地打包，不能可信交付 |
| `10` | 高层自动化门禁接管主验证责任 | 发布前仍严重依赖人工验收 |
| `11` | 10k+ 规模性能基线、热点治理、回归机制闭环 | 性能优化仍无量化证明 |
| `12` | revision 模型、diff、恢复和回滚语义闭环 | 历史只能看不能恢复 |
| `13` | 发布 go/no-go、能力兑现清单、下一轮输入冻结 | 无法区分“已实现”和“目标态” |

默认验收等级要求：

- 所有 Step 的默认完成等级必须达到 [`95-架构能力闭环验收标准`](./95-架构能力闭环验收标准.md) 中的 `L4`
- 任何仅达到 `L1-L3` 的 Step，都只能视为“已设计 / 已实现 / 已验证”，不能标记为“已完成”

## 5. 实施波次

### 波次 A：基线、骨架与安全前置

- `00`
- `01`
- `02`
- `03`

目标是把“文档清楚但执行口径不统一”的状态，推进到“执行门禁冻结、骨架明确、会话和配置风险开始收敛”的状态。

### 波次 B：工作区主链路与本地基础能力

- `04`
- `05`
- `06`

目标是完成数据访问、初始化、编辑保存和本地草稿恢复三条主链路，使后续搜索和同步有稳定承载层。

### 波次 C：搜索、同步与桌面交付

- `07`
- `08`
- `09`

目标是完成本地检索、同步队列、冲突恢复、桌面能力治理和可信发布链一期。

### 波次 D：质量、性能、版本与最终收口

- `10`
- `11`
- `12`
- `13`

目标是把系统从“可开发、可演示”推进到“可验证、可回归、可交付、可持续迭代”。

### 5.1 最速并行执行建议

如果目标是在风险可控前提下最快完成全部 Step，建议采用以下策略：

| 执行规模 | 推荐编组 | 推荐节奏 | 说明 |
| --- | --- | --- | --- |
| `2` 人/Agent | `A=主脊柱+集成` `B=功能车道` | 只在 `02/03`、`05/06`、`07/09`、`11/12` 做有限并行 | 最稳妥，适合高确定性推进 |
| `4` 人/Agent | `A=集成/文档` `B=壳层/安全` `C=工作区/本地层` `D=搜索/同步/测试` | 每波次中期集成一次，波次末总验收一次 | 当前仓库的推荐最优解 |
| `6` 人/Agent | `A=集成/架构` `B=壳层/安全` `C=初始化/编辑/本地层` `D=搜索/同步` `E=桌面/发布链` `F=测试/性能/版本历史` | 周三中期集成，周五审计收口 | 速度最快，但必须严格执行 `94/96/98` |

最速执行的硬规则：

- 不允许跳过 `00 -> 01 -> 02 -> 03` 主脊柱顺序
- `10` 必须从 `03` 后尽早启动，作为横向质量车道贯穿执行
- 任何波次都必须保留独立验证车道，不与功能车道混写
- 只有前一波次完成总验收，下一波次才能全面放行

## 7. 每一步必须产出的内容

每个 step 文档都必须明确：

- 执行输入
- 本步非目标
- 最小输出
- 推荐 review 产物
- 推荐并行车道
- 架构能力闭环判定
- 快速并行执行建议
- 完成后必须回写的架构文档

- 实施落地规划
- 测试计划
- 结果验证
- 检查点

任何没有这些内容的 step，都视为未达执行标准。

## 8. 执行证据要求

每个 step 完成时，必须同步产出以下证据中的适用项：

- `docs/review/` 中的问题清单、决策记录、复盘记录
- 新增或更新的 `docs/架构/` 文档
- 对应代码目录下的测试、基准和脚本
- `sdkwork-canvas-pc-react/scripts/*` 或 workflow 的验证结果
- 可重复执行的验证命令及结果摘要

推荐证据分层如下：

- 架构类证据：`docs/架构/`
- 执行类证据：`docs/step/`
- 审计与复盘类证据：`docs/review/`
- 自动化验证类证据：测试、脚本、workflow contract
- 临时运行证据：`tmp/` 下的局部验证产物

## 9. 极致打磨规则

后续所有 step 在执行和复盘时，必须额外通过以下九个维度自查：

- 架构对齐是否清晰到具体文档编号
- 代码触达范围是否精确到包、目录和高风险文件
- 数据、会话、平台能力边界是否明确
- 测试是否覆盖单元、集成、E2E、视觉、性能或脚本验证中的适用项
- 结果验证是否可重复，不依赖人工口头说明
- 检查点是否具备“完成 / 未完成”的明确判定标准
- 风险与回滚是否真实可执行，而不是泛泛而谈
- 串行依赖与可并行车道是否边界清楚
- 是否形成 `docs/review/` 中的审计和复盘沉淀

建议在每次 step 收尾时，结合 [`91-Step质量审计清单与复盘模板`](./91-Step质量审计清单与复盘模板.md) 打分，目标不低于 `90/100`。

## 10. 当前阶段重点约束

结合当前仓库现状，执行时必须额外遵守以下约束：

- 不允许把本地优先、搜索、同步、版本历史直接塞进 `@sdkwork/canvas-canvas` 页面组件
- 不允许让 Web 与 Desktop 分叉两套业务壳
- 不允许继续长期依赖 `localStorage` 保存高敏会话而不做迁移方案
- 不允许先做“目标态宣传”再补测试和回滚
- 新增本地能力层时必须先定义存储权威源和同步边界
- 搜索、同步、版本、updater 等新能力必须有独立包或独立清晰模块落点

## 11. 当前仓库执行起点

基于当前已知现状，执行计划的真实起点是：

- 当前主工程是 `sdkwork-canvas-pc-react`
- 当前已有 `canvas-shell / canvas-auth / canvas-canvas / canvas-user / canvas-core / canvas-desktop` 等清晰包边界
- 当前最大短板是本地优先、全文搜索、同步引擎、版本历史、可信发布链和高层自动化测试
- 当前桌面壳、启动链、认证编排和笔记工作区已经具备高质量基础，不需要推倒重来
- 当前执行重点不是重做技术栈，而是在现有栈上补齐能力层和治理层

## 12. 使用方式

推荐使用顺序：

1. 先阅读 [`00-总实施原则与执行门禁`](./00-总实施原则与执行门禁.md)
2. 再执行 [`01-现状基线冻结与差距审计`](./01-现状基线冻结与差距审计.md)
3. 执行前使用 [`90-架构能力-Step-代码目录-证据映射矩阵`](./90-架构能力-Step-代码目录-证据映射矩阵.md) 做能力对照
4. 高风险 step 实施前，使用 [`92-Step输入输出与阻塞升级规则`](./92-Step输入输出与阻塞升级规则.md) 形成执行卡
5. 若同一波次需要提速，先使用 [`94-Step并行执行编排与车道拆分建议`](./94-Step并行执行编排与车道拆分建议.md) 确定并行车道，再使用 [`96-Step并行执行周计划与排班建议`](./96-Step并行执行周计划与排班建议.md) 与 [`98-Step并行车道交付包与集成交接标准`](./98-Step并行车道交付包与集成交接标准.md) 固化节奏和交接包
6. 若目标是最快且完整交付，优先按 [`99-Step极速并行执行总作战手册`](./99-Step极速并行执行总作战手册.md) 选择 `Fast-S / Fast-M / Fast-L` 执行模式
7. 每完成一个 step，使用 [`91-Step质量审计清单与复盘模板`](./91-Step质量审计清单与复盘模板.md)、[`95-架构能力闭环验收标准`](./95-架构能力闭环验收标准.md) 和 [`97-Step完成后的架构回写与能力兑现清单`](./97-Step完成后的架构回写与能力兑现清单.md) 做自检、闭环判定和架构回写
8. 每完成一个波次，使用 [`93-波次里程碑与阶段总验收矩阵`](./93-波次里程碑与阶段总验收矩阵.md) 做总验收
9. 最后按 [`13-发布就绪与持续迭代闭环`](./13-发布就绪与持续迭代闭环.md) 完成发布就绪和下一轮输入沉淀

如果执行中出现重大架构偏移，必须先回写 `docs/架构/` 和 `docs/step/`，再继续编码。
