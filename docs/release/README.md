# Release 文档规范

本目录用于沉淀 `sdkwork-canvas` 的版本管理、变更日志、发布说明和发布候选结论，确保每一轮 Step 实施结果都具备可追溯、可审计、可发布的证据链。

## 1. 目录目标

- 记录版本号、发布日期、候选状态与发布状态
- 累积专业 changelog
- 沉淀重要版本或里程碑版本说明
- 作为 `docs/step/`、`docs/review/`、`docs/架构/` 与实际交付物之间的发布侧证据

## 2. 最低维护要求

至少维护以下文件：

- `CHANGELOG.md`：全局累计变更日志
- `最新版本说明.md`：当前版本、候选版本、发布日期、发布状态、已知风险、回滚结论

在需要对某次版本进行完整归档时，可新增：

- `YYYY-MM-DD-vX.Y.Z-发布说明.md`

## 3. 版本规则

- 采用语义化版本：`MAJOR.MINOR.PATCH`
- 在尚未形成正式候选版本前，可在 `CHANGELOG.md` 顶部保留 `Unreleased` 区域；一旦形成候选发布版本，必须落为具体语义化版本
- 所有日期统一使用绝对日期格式：`YYYY-MM-DD`
- 任何真实的代码、测试、文档、脚本、构建、CI、发布调整，都必须进入 changelog
- 版本号提升必须与实际影响范围匹配，不能随意跳号，也不能长期不升版本

## 4. Changelog 书写要求

推荐分类如下：

- `Added`
- `Changed`
- `Fixed`
- `Refactored`
- `Performance`
- `Security`
- `Tests`
- `Docs`
- `Build`
- `CI`
- `Release`

每条变更建议至少说明：

- 影响范围
- 改动目的
- 风险控制方式
- 验证结果摘要
- 关联 Step / 波次 / 架构能力

禁止使用以下模糊描述：

- “优化若干”
- “修复一些问题”
- “调整细节”
- “完善体验”

应改为可审计表述，例如明确到模块、能力、验证结论和风险控制方式。

## 5. 与其他目录的关系

- `docs/step/` 定义“要做什么、按什么顺序做、什么算完成”
- `docs/架构/` 定义“为什么这样做、目标边界是什么、能力如何设计”
- `docs/review/` 记录“这一轮实际上做了什么、验证了什么、还差什么”
- `docs/release/` 记录“这些结果如何形成版本、如何对外发布、如何追溯和回滚”

四者必须保持一致，不能出现代码已经变化、review 已确认、但 release 记录缺失的情况。
