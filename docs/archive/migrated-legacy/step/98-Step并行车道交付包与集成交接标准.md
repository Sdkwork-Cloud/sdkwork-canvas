# 98. Step 并行车道交付包与集成交接标准

## 1. 文档定位

本文件用于规定：当一个 Step 由多个并行车道或多个子 Agent 同时推进时，每个车道在交给集成 Owner 之前，必须提交什么内容、满足什么标准、如何交接，才能安全进入统一集成。

目标是降低并行开发的隐性成本，避免“代码写完了，但无法安全合并”。

## 2. 适用场景

适用于以下情况：

- 多个子团队同时推进同一波次
- 多个子 Agent 并行执行不同车道
- 功能车道、测试车道、文档车道分开推进

## 3. 交付包定义

每个并行车道在收口时，必须形成一份“交付包”。交付包不是单纯的代码 diff，而是最小可集成单元。

一个完整交付包应包含：

- 变更范围说明
- 主写入目录清单
- 非写入边界说明
- 设计与约束摘要
- 测试与验证结果
- 风险与回滚
- 需要集成 Owner 协调的问题

## 5. 可集成标准

车道交付包只有同时满足以下条件，才能进入集成：

### 5.1 边界清晰

- 主写入目录明确
- 不写入的边界明确
- 共享文件 Owner 清晰

### 5.2 设计清晰

- 设计摘要足以让集成 Owner 理解目标和约束
- 与其他车道的接口点清晰

### 5.3 测试充分

- 本车道适用测试已执行
- 结果有摘要
- 已明确未覆盖项与影响范围

### 5.4 风险可控

- 已说明风险
- 已提供回滚方式
- 无未登记的高风险共享文件冲突

## 6. 不可集成信号

出现以下任一情况，交付包不得进入统一集成：

- 没有主写入目录说明
- 关键共享文件有多头主写
- 无测试结果
- 风险和回滚为空白
- 设计目标与当前 Step 目标不一致
- 需要依赖其他车道，但依赖关系未说明

## 7. 集成交接流程

推荐流程如下：

1. 各车道先提交交付包
2. 集成 Owner 先审交付包，再审代码
3. 确认共享文件 Owner 和合并顺序
4. 做统一集成验证
5. 更新 `docs/review/` 集成记录
6. 通过后再判定 Step 是否完成

## 9. 多子 Agent 集成附加规则

### 9.1 子 Agent 不得做的事

- 不得擅自改写其他车道主写目录
- 不得在未声明的情况下大范围清理共享文件
- 不得跳过测试和验证直接提交“已完成”结论

### 9.2 主 Agent 必做的事

- 校验每个交付包的边界
- 审核共享文件冲突
- 统一做最终验证
- 统一更新文档与结论

## 10. 推荐交付包内容粒度

好的交付包应具备：

- 足够独立，能被单独审查
- 足够完整，能支撑集成
- 足够克制，不跨越多个主目标

不好的交付包通常表现为：

- 改动面极大但解释极少
- 设计和测试为空
- 把多个能力混在一起

## 11. 与其他治理文档的关系

- `90`：定义能力归属与代码落点
- `91`：定义质量审计标准
- `92`：定义输入输出和阻塞升级
- `94`：定义并行车道如何拆
- `96`：定义每周如何排班
- `97`：定义完成后如何回写
- `98`：定义车道如何安全交付给集成 Owner

## 12. 结论

并行执行真正的难点不在“能不能同时写代码”，而在“能不能稳定交接并集成”。本文件的作用，就是把并行执行的最后一公里标准化，减少返工、冲突和不可控风险。
