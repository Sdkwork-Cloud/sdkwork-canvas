# 06-业务流程-应用接口与集成设计-认证宿主边界补充

## 1. 背景

在 `2026-04-13` 的回归止损中，根级 `workspace-boundary-contract` 发现旧合同仍在要求仓库外 `@sdkwork/auth-pc-react` 保持“零 `@sdkwork/core-pc-react` 依赖”。但当前真实上游源码已经把 `@sdkwork/core-pc-react` 作为 shared auth 的默认 runtime 依赖，用于默认的会话与 app client 访问。

这意味着 canvas 仓库的权威边界应当从“约束上游 shared auth 的内部依赖结构”切换为“约束 canvas 仓库自己的认证宿主层边界”。

## 2. 当前真实边界

### 2.1 宿主边界

- `@sdkwork/canvas-pc-auth` 是 canvas 仓库唯一允许直接接入 `@sdkwork/auth-pc-react` 的宿主包。
- 其它业务包、壳层包、页面包只能依赖 `@sdkwork/canvas-pc-auth`，不得直接导入 `@sdkwork/auth-pc-react`。

### 2.2 运行时边界

- shared auth 包可以保留自己的默认 runtime。
- 但在 canvas 仓库内，真实生效的运行时必须由 `packages/sdkwork-canvas-pc-auth/src/services/sdkworkAuthBridge.ts` 重新绑定：
  - `getClient -> canvas-core`
  - `persistSession -> canvas-core`
  - `readSession -> canvas-core`
  - `clearSession -> canvas-core`
  - `resolveAccessToken -> canvas-core`

### 2.3 会话权威边界

- `canvas-core` 仍然是 canvas 仓库会话与 app-sdk client 的权威宿主。
- `canvas-auth` 是认证体验与共享 auth 控制器的宿主适配层。
- `@sdkwork/auth-pc-react` 在 canvas 内被视为共享能力源，而不是最终运行时权威层。

## 3. 设计收益

### 3.1 降低仓库外漂移对主门禁的误伤

- 根级质量门禁不再直接依赖仓库外包的内部依赖树细节。
- 质量门禁改为约束本仓库真正可控的宿主边界与桥接事实。

### 3.2 保持会话权威源不漂移

- 即使 shared auth 默认 runtime 演进，canvas 仓库仍通过 `sdkworkAuthBridge` 把会话写回、读取、清理与 access token 解析固定到 `canvas-core`。

### 3.3 防止认证接入继续散落

- 通过合同冻结“只有 `canvas-auth` 可以直接接入 shared auth”，避免业务包重新出现直连 shared auth 的边界回退。

## 4. 验收标准

设计标准：

- `canvas-auth` 作为唯一 shared auth 宿主边界，定义清晰。
- `canvas-core` 继续担任会话与 app-sdk 运行时权威源。

实现标准：

- `packages/sdkwork-canvas-pc-auth/package.json` 不直接依赖 `@sdkwork/core-pc-react`。
- `sdkworkAuthBridge.ts` 显式覆盖 shared auth 的默认 runtime 入口。

测试标准：

- `workspace-boundary-contract.test.mjs` 必须验证：
  - `canvas-auth` 没有 `core-pc-react` 直接依赖
  - `sdkworkAuthBridge.ts` 仍绑定 `canvas-core`
  - 除 `canvas-auth` 外没有其它 TypeScript 包直连 `@sdkwork/auth-pc-react`

集成标准：

- 根级 `pnpm.cmd typecheck` 可完整通过，不再被旧 shared auth 包结构假设阻断。

## 5. 当前结论

- 认证宿主边界已重新对齐为“`canvas-auth` 宿主 + `canvas-core` 权威 runtime”。
- `Step 03` 的当前有效等级继续保持为 `L4`。
- 当前纠偏不改变 `Step 05` 的执行入口；后续应回到保存链可靠性主线继续推进。
