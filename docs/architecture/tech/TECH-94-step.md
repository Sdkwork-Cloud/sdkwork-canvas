# 94. Step 并行执行编排与车道拆分建议

## 1. 文档定位

本文件用于明确 `docs/step/00-13` 中哪些 Step 必须串行推进，哪些 Step 可以在波次内部并行推进，以及并行时应如何按“主写入范围”而不是按“概念主题”拆分车道。

目标不是让所有工作并行，而是在不破坏架构边界和质量门禁的前提下，尽可能快、尽可能安全地完成重构与升级。

## 2. 总体执行原则

推荐总模式：

`串行主脊柱 + 波次内有限并行 + 独立验证车道 + 统一集成收口`

含义如下：

- Step 级别保持架构依赖顺序
- 同一波次内按写入范围拆分多个并行车道
- 验证尽量由独立车道负责，不与实现车道混写
- 最终由集成 Owner 做统一收口与发布判断

## 3. 必须串行执行的 Step

以下 Step 不允许在 Step 级跳过前置顺序：

| Step | 必须串行原因 |
| --- | --- |
| `00` | 不先冻结执行门禁，后续所有车道口径都会漂移 |
| `01` | 不先完成现状基线与差距审计，后续并行车道容易各自理解“目标” |
| `02` | 不先冻结共享壳层和工程骨架，后续能力无稳定落点 |
| `03` | 不先冻结会话安全与配置治理，工作区、本地层、桌面链都可能返工 |
| `13` | 最终发布就绪与闭环必须在前面全部能力完成后统一执行 |

说明：

- 这些 Step 内部可以拆并行车道
- 但 Step 之间不得越级并行

## 4. 可串行优先、有限交叠的 Step

以下 Step 可以在满足前置条件后进行小范围交叠，但不得失去主依赖顺序：

| Step | 前置 | 允许交叠 | 说明 |
| --- | --- | --- | --- |
| `04` | `02` `03` | 可与 `10` 的测试基础设施预埋交叠 | 工作区主链路应先行，测试车道可提前铺路 |
| `05` | `04` | 可与 `10` 局部交叠 | 编辑器可靠性需要依赖初始化链基本稳定 |
| `06` | `04` `05` | 可与 `07` 设计准备交叠，不可与 `08` 实现重叠过深 | 本地层是搜索和同步的承载基础 |
| `07` | `06` | 可与 `09` 的桌面发布链部分工作并行 | 搜索与桌面壳写入范围天然相对独立 |
| `08` | `06` `07` | 可与 `09` `10` `11` 局部并行 | 同步完成最小状态机后，可启动性能和桌面侧配套 |
| `09` | `03` `08` | 可与 `10` `11` 并行 | 桌面壳与发布链可单独车道推进 |
| `10` | `03` 后可启动，`13` 前必须完成 | 可跨多个波次持续推进 | 质量门禁是全程护栏 |
| `11` | `07` `08` `10` | 可与 `12` 部分并行 | 性能与历史恢复都基于更稳定的数据与测试底座 |
| `12` | `06` `08` `10` | 可与 `11` 部分并行 | 版本历史依赖本地层、同步与测试基座 |

## 5. 波次级并行结论

### 5.1 `Wave-A` 并行策略

包含：

- `00`
- `01`
- `02`
- `03`

推荐顺序：

1. `00` 完成后再启动 `01`
2. `01` 审计结论冻结后，再并行推进 `02` 与 `03` 的局部工作
3. `02` 与 `03` 集成收口后，才能进入 `Wave-B`

推荐车道：

- `A1-审计与文档车道`
  - 主写入：`docs/step` `docs/review` `docs/架构`
  - 责任：差距矩阵、术语统一、执行门禁回写
- `A2-骨架收敛车道`
  - 主写入：`sdkwork-canvas-pc-react/packages/sdkwork-canvas-shell` `sdkwork-canvas-pc-react/packages/sdkwork-canvas-desktop`
  - 责任：共享壳层、目录落点、包边界
- `A3-安全与配置车道`
  - 主写入：`sdkwork-canvas-pc-react/packages/sdkwork-canvas-auth` `sdkwork-canvas-pc-react/packages/sdkwork-canvas-core` `sdkwork-canvas-pc-react/scripts`
  - 责任：会话安全、配置治理、迁移路径
- `A4-验证车道`
  - 主写入：测试目录、`docs/review`
  - 责任：contract test、启动冒烟、安全验证

### 5.2 `Wave-B` 并行策略

包含：

- `04`
- `05`
- `06`

推荐顺序：

1. `04` 主链路先形成基本稳定初始化编排
2. 再拆 `05` 与 `06` 进入并行
3. `05` 和 `06` 集成后再启动更深层搜索/同步实现

推荐车道：

- `B1-初始化编排车道`
  - 主写入：`sdkwork-canvas-pc-react/packages/sdkwork-canvas-canvas` `sdkwork-canvas-pc-react/packages/sdkwork-canvas-core`
  - 责任：repository/store/init orchestration
- `B2-编辑可靠性车道`
  - 主写入：`sdkwork-canvas-pc-react/packages/sdkwork-canvas-canvas`
  - 责任：编辑器事件流、保存状态机、失败恢复
- `B3-本地存储车道`
  - 主写入：`sdkwork-canvas-pc-react/packages/sdkwork-canvas-local` `sdkwork-canvas-pc-react/packages/sdkwork-canvas-desktop`
  - 责任：本地 schema、离线草稿、崩溃恢复
- `B4-测试与回归车道`
  - 主写入：测试目录、脚本、`docs/review`
  - 责任：集成测试、断网/崩溃回归、验证记录

### 5.3 `Wave-C` 并行策略

包含：

- `07`
- `08`
- `09`

推荐顺序：

1. `07` 先冻结索引模型和检索语义
2. `08` 基于本地层与索引结果推进同步队列和冲突恢复
3. `09` 可与 `08` 并行推进桌面 runtime 与发布链

推荐车道：

- `C1-搜索索引车道`
  - 主写入：`sdkwork-canvas-pc-react/packages/sdkwork-canvas-search` `sdkwork-canvas-pc-react/packages/sdkwork-canvas-canvas`
  - 责任：索引构建、查询语义、命令面板共享入口
- `C2-同步引擎车道`
  - 主写入：`sdkwork-canvas-pc-react/packages/sdkwork-canvas-sync` `sdkwork-canvas-pc-react/packages/sdkwork-canvas-local`
  - 责任：队列、重试、冲突恢复、离在线切换
- `C3-桌面壳与发布链车道`
  - 主写入：`sdkwork-canvas-pc-react/packages/sdkwork-canvas-desktop` `.github/workflows` `sdkwork-canvas-pc-react/scripts`
  - 责任：桌面 runtime、权限能力、可信交付、dry-run
- `C4-验证与基准车道`
  - 主写入：测试目录、基准脚本、`docs/review`
  - 责任：搜索准确性、同步冲突演练、bundle 验证

### 5.4 `Wave-D` 并行策略

包含：

- `10`
- `11`
- `12`
- `13`

推荐顺序：

1. `10` 作为贯穿式车道持续建设
2. `11` 与 `12` 在满足前置后并行
3. `13` 统一收口

推荐车道：

- `D1-质量门禁车道`
  - 主写入：测试目录、workflow、脚本
  - 责任：单测/集成/E2E/视觉/性能/安全门禁
- `D2-性能治理车道`
  - 主写入：`sdkwork-canvas-pc-react/packages/sdkwork-canvas-canvas` `sdkwork-canvas-pc-react/packages/sdkwork-canvas-search`
  - 责任：10k+ 工作区性能优化、基准与回归
- `D3-版本历史车道`
  - 主写入：`sdkwork-canvas-pc-react/packages/sdkwork-canvas-history` `sdkwork-canvas-pc-react/packages/sdkwork-canvas-local` `sdkwork-canvas-pc-react/packages/sdkwork-canvas-canvas`
  - 责任：revision、diff、恢复交互
- `D4-发布收口车道`
  - 主写入：`docs/架构` `docs/step` `docs/review` `.github/workflows`
  - 责任：阶段验收、发布候选、回滚结论、下一轮 backlog

## 6. 车道拆分的硬规则

### 6.1 按主写入范围拆分

允许：

- 搜索车道主写 `canvas-search`
- 同步车道主写 `canvas-sync`
- 桌面车道主写 `canvas-desktop`

不允许：

- 搜索和同步同时深度主写同一组 store 文件
- 编辑可靠性和本地存储同时重写同一保存状态机
- 发布链车道顺手修改工作区业务逻辑

### 6.2 共享边界文件必须单 Owner

以下类型文件必须指定唯一 Owner：

- 顶层 Provider 组合
- 权威 store 类型定义
- 同步状态模型
- 搜索索引公共接口
- 桌面 runtime 入口
- 核心 workflow 文件

### 6.3 验证车道尽量独立

验证车道应优先负责：

- 脚本
- 测试
- 基准
- 审计记录

避免与功能车道混在同一批次无边界推进。

## 7. 子 Agent 并行适配建议

如果采用多子 Agent 并行，建议遵循以下原则：

### 7.1 适合并行交给子 Agent 的工作

- 独立包内的能力实现
- 测试和脚本补齐
- 文档回写和审计记录初稿
- 与共享边界解耦的性能优化

### 7.2 不适合直接并行交给子 Agent 的工作

- 共享状态模型和权威类型的最终拍板
- 发布候选和 go/no-go 结论
- 跨多个高风险目录的大规模同时改造
- 没有清晰写入范围的“先探索一下”

### 7.3 子 Agent 交付要求

- 必须说明主写入目录
- 必须说明不写入的边界
- 必须附测试和验证结果
- 必须给出风险与回滚说明

## 8. 推荐的最快执行路径

在当前 `sdkwork-canvas` 仓库现状下，建议的最快路径如下：

1. 串行完成 `00 -> 01`
2. 在 `Wave-A` 内并行推进 `02` 骨架车道、`03` 安全车道、验证车道
3. 进入 `Wave-B` 后，让初始化车道先行，随后编辑可靠性与本地存储并行
4. 进入 `Wave-C` 后，让搜索、同步、桌面与验证四车道并行
5. 进入 `Wave-D` 后，让质量门禁持续运行，性能与版本历史并行，最后统一 `13` 收口

这是当前速度、风险和质量的平衡最优解。

## 9. 并行风险清单

- 多车道同时修改同一高风险 store
- 验证车道滞后，导致问题晚发现
- 文档没有及时回写，导致后续车道基于旧假设继续开发
- 功能实现完成但未满足 `98` 交付包标准，集成成本急剧上升

### 9.1 并行放行前必须冻结的四件事

任何一个波次开始大规模并行前，都必须先冻结以下四项：

1. 本波次主写入目录与单一 Owner
2. 共享边界文件的最终拍板人
3. 统一测试口径与验证命令
4. 失败时的回滚或降级路径

四项里任意一项未冻结，都不应放行多车道并行。

## 10. 放行与总算法

### 10.1 波次放行信号

为避免“看起来可以并行，实际上前置还没稳定”，推荐以下放行信号：

| 放行对象 | 必须先满足的信号 |
| --- | --- |
| `02/03` 并行 | `01` 差距矩阵和风险优先级已冻结 |
| `05/06` 并行 | `04` 的 `repository / orchestration / selector` 边界已冻结 |
| `08` 启动 | `06` 权威源定义已冻结，`07` 至少完成索引 schema 冻结 |
| `09` 深入推进 | `03` 安全治理稳定，`08` 已完成最小同步状态机 |
| `11/12` 并行 | `10` 已能提供稳定回归框架，`08` 已具备最小一致性模型 |
| `13` 收口 | `00-12` 已形成证据，不再存在 `B3/B4` 阻塞 |

### 10.2 单轮并行执行总算法

推荐每轮都按以下顺序运行：

1. 冻结本轮目标、边界、Owner、检查点
2. 分发车道，严格按主写入目录执行
3. 周中或中期做一次集成检查，只处理冲突和偏移
4. 统一执行验证车道，不允许只验证局部
5. 输出审计、复盘和回写
6. 只有通过 `91/93/95/98` 后，才进入下一轮

## 11. 结论

高质量并行不是“大家一起改”，而是：

- 串行冻结主脊柱
- 波次内按写入范围拆车道
- 把验证独立出来
- 用统一集成 Owner 收口

后续所有 Step 的多子 Agent 执行建议，都应以本文件为统一编排依据。
