# Step 05 失败反馈与重试状态机审计

- 日期：`2026-04-07`
- 阶段：`Step 05 / L2`
- 波次：`Wave-B / 第二十七轮推进`
- 本轮主题：`保存失败反馈与重试状态机`

## 1. 本轮目标

本轮承接上一轮已经完成的“统一刷盘入口”能力，继续解决保存链中的第三个关键结构问题：保存失败后虽然可以再次触发 `persistActiveNote()`，但 store、编辑器、错误横幅与国际化文案之间并没有形成同一份显式状态机与用户反馈模型。

本轮聚焦的真实缺口如下：

1. `saveState` 只区分 `saving / saved / error`，无法准确表达“失败后的重试中”和“重试成功后已恢复”。
2. `NoteEditorPane.tsx` 与 `NotesWorkspacePage.tsx` 分别依赖零散的 `saveState` 判断，保存反馈逻辑没有统一模型。
3. 错误横幅虽然能显示失败信息，但没有与 store 状态机对齐的重试动作入口。
4. `zh-CN` 未补齐新的保存反馈文案，中文环境会退回英文键值或缺失翻译。

## 4. 架构判断

本轮完成后，保存失败反馈链的职责边界稳定为：

1. `noteWorkspaceSaveFeedback.ts`
   - 负责保存反馈模型与状态机迁移规则。
   - 不关心 DOM 绑定，也不直接发起持久化。
2. `useNotesWorkspaceStore.ts`
   - 负责保存链真实执行与状态落地。
   - 只消费状态迁移 helper，不再内嵌重试状态语义。
3. `NoteEditorPane.tsx`
   - 只消费统一反馈模型。
   - 不再自行拼装“按钮是否可点、是否 loading、状态文案 key 是什么”。
4. `NotesWorkspacePage.tsx` 与 `NotesWorkspaceErrorBanner.tsx`
   - 共同构成“错误可见 + 重试命令暴露”的 UI 边界。
   - 页面只在 `retryAvailable` 为真时暴露 `flushDraft`。

这意味着 Step 05 已完成第三条主链事实：保存失败不再只是一个孤立的 `error` 标记，而是具备“错误 -> 重试中 -> 恢复”的显式可见状态机。

## 5. 已控制风险

1. `error` 状态不再是“只显示失败但语义不完整”的半闭环状态。
2. 编辑器保存按钮、状态徽标与错误横幅不再各自维护不同的保存语义。
3. 中文环境的新增保存反馈文案已补齐，不会在商业化交付场景中出现英文回退。
4. 新 contract 已进入 `test:workspace:contracts` 主链，后续若有人移除 `retrying / recovered` 或把重试按钮改回页面本地散落逻辑，会在总门禁中直接暴露。

## 6. 仍未完成项

1. 当前仍没有真正的 save queue，连续失败或并发保存尚未进入串行排队模型。
2. 重试策略仍然是“用户触发立即重试”，还没有自动退避、最大重试次数与观测事件。
3. 切换笔记、危险操作、页面关闭前三类高风险场景的 flush 证据矩阵仍未闭环。
