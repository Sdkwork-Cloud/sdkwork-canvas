## 当前用户可见变化

- 当主阻塞同步问题可以定位到具体笔记时，工作区同步卡片不再继续展示误导性的“重试同步”，而是切换为“查看受影响笔记”。
- 当当前没有可直接定位的阻塞笔记，但队列里仍有 `pending` 任务时，页面继续保留“重试同步”入口，用于请求已有 runtime 继续 drain。
- “最新问题”现在优先展示更可读的失败或冲突消息，而不是只暴露原始错误码。
- 页面层不再把所有同步问题都收敛成一个 drain 动作，用户可以更快进入受影响笔记做人工处理。

## 为什么这轮仍然只算 L3

- 这轮交付的是“恢复动作语义更诚实、受影响笔记可定位”，不是“真实远端恢复闭环”。
- `review-note` 的真实语义只是“打开受影响笔记作为人工恢复入口”，它不会：
  - 解决远端冲突
  - 重写 `failed / conflict` 任务状态
  - 合并 `remoteCursor`
- `requestSyncDrain()` 仍然只会请求当前 runtime 继续 drain 已可执行队列；它不会让 `failed / conflict` 任务 magically 变成已恢复。
- 当前仍未完成：
  - 真实 `remoteApply`
  - ack apply / `remoteCursor` 合并闭环
  - 真实 conflict recovery UI
  - 离线/在线切换 smoke
