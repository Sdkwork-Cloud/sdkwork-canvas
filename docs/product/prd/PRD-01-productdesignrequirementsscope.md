# 01. 产品设计与需求范围

## 1.1 文档目的

本文用于回答四个问题：

1. `sdkwork-canvas` 当前到底是什么产品。
2. 当前仓库已经实现了哪些真实能力。
3. 行业领先笔记应用通常具备哪些能力标准。
4. SDKWork Canvas 应把哪些能力定义为 P0、P1、P2，避免范围失控。

本文的对象是当前主交付物 `sdkwork-canvas-pc-react`，时间基线为 `2026-04-07`。

---

## 1.2 当前产品定位

从仓库结构、路由、桌面壳、笔记工作区和账号设置能力来看，当前产品定位不是轻量便签，也不是文档平台的子模块，而是：

**面向专业知识工作者、内容创作者和开发者的桌面优先笔记工作台。**

其核心特征如下：

- 以桌面应用为主战场，强调窗口控制、托盘、快捷键和工作区效率。
- 运行时和发布拓扑不是“纯桌面单入口”，而是共享业务壳的 Web/Desktop 双入口；桌面体验是主战场，Web 入口仍是正式发布拓扑的一部分。
- 认证入口已经具备多方式身份能力，支持密码、短信验证码、邮箱验证码、二维码登录与 OAuth 登录编排。
- 支持认证、个人设置、富文本编辑、分类整理、收藏、回收站等完整单人笔记闭环。
- 已经具备明显的“专业工作台”结构，而不是单列表单页。
- 当前更接近“云端驱动的桌面知识工作台”，尚不是“本地优先 + 多端同步 + 协作平台”。

### 1.2.1 当前平台拓扑

| 维度 | 当前事实 |
|---|---|
| 共享业务壳 | Web 与 Desktop 最终都收敛到 `@sdkwork/canvas-pc-shell` 的 `AppRoot` |
| Web 入口 | `sdkwork-canvas-pc-react/src/main.tsx`，负责 `ensureI18n()` 后渲染 `App`，而 `App` 当前直接返回 `AppRoot` |
| Desktop 入口 | `packages/sdkwork-canvas-pc-desktop/src/main.tsx`，通过 `createDesktopApp()` 增加启动外观快照、Tauri runtime 探测、桌面桥注入与启动屏 |
| 发布元数据 | `sdkwork.app.config.json` 同时声明 `WEB / DESKTOP / DESKTOP_WINDOWS / DESKTOP_MACOS / DESKTOP_LINUX` |
| 默认包 | 当前 `defaultPackageId = web-production`，说明元数据层保留 Web bundle 作为默认安装包语义 |
| 当前结论 | 产品定位是“桌面优先”，但发布与运行时拓扑不是“只有桌面，没有 Web” |

## 1.3 行业领先能力基线

结合 Notion、Obsidian、Apple Notes、OneNote、Craft 的官方能力信息，可抽象出行业领先笔记产品的共同基线：

| 维度 | 行业领先能力 | 代表产品特征 |
|---|---|---|
| 内容组织 | 文档、页面、文件夹、标签、关系、引用、空间/工作区 | Notion、Craft、Obsidian |
| 编辑体验 | 富文本、Markdown/块编辑、任务列表、代码块、快捷键、命令面板 | Notion、Obsidian、OneNote |
| 知识发现 | 全文检索、标签检索、双向链接、图谱、最近访问、智能推荐 | Obsidian、Notion |
| 同步能力 | 多端同步、冲突处理、离线可用、历史版本、恢复 | Apple Notes、Notion、Craft |
| 协作分享 | 评论、共享、权限、只读发布、团队空间 | Notion、OneNote、Craft |
| 桌面体验 | 原生窗口语义、系统托盘、快捷操作、稳定启动、自动更新 | Apple Notes、Craft、Electron/Tauri 类桌面应用 |
| 安全治理 | 会话保护、最小权限、审计、恢复、可信发布 | 企业级产品通用要求 |
| 工程成熟度 | 监控、测试、回滚、质量门禁、发布渠道治理 | 头部商用产品通用要求 |

对 SDKWork Canvas 来说，不应机械复制所有产品形态，而应抽取最匹配的能力组合：

- 用 Notion 的“结构化工作台和协作模型”作为上限参考。
- 用 Obsidian 的“本地优先、知识发现、扩展能力”作为能力上限参考。
- 用 Apple Notes 的“低摩擦、快速稳定、系统整合”作为体验基准。
- 用 Craft 的“设计感、卡片化结构、跨设备内容体验”作为设计质量基准。

## 1.4 目标用户与关键场景

### 1.4.1 目标用户

| 用户类型 | 关键诉求 | 当前支持度 | 未来目标 |
|---|---|---:|---|
| 个人知识工作者 | 快速捕获、归档、检索、沉浸式编辑 | 高 | 提升为离线优先和全局检索 |
| 内容创作者 | 草稿整理、结构化写作、发布状态管理 | 高 | 增加版本历史和多端编辑 |
| 开发者 | 代码笔记、任务跟踪、技术资料管理 | 中高 | 增加双向链接、代码片段治理 |
| 小团队成员 | 文档共享、协同编辑、审计与权限 | 低 | 未来重点突破方向 |
| 企业知识管理员 | 空间治理、访问控制、发布与合规 | 低 | 作为平台化阶段目标 |

### 1.4.2 当前高频场景

- 登录进入桌面工作区，继续上次编辑状态。
- 创建文档、文章或代码型笔记。
- 通过命令面板快速创建、切换笔记、切换视图或跳转账户页。
- 通过文件夹、收藏、最近、搜索快速定位笔记，并通过拖拽重组层级。
- 在富文本编辑器中持续编辑并自动保存。
- 调整主题、语言、窗口布局和账户资料。
- 通过托盘恢复主窗口，快速进入笔记或账户页。

### 1.4.3 行业领先目标场景

- 多设备间无感同步和离线恢复。
- 10k+ 笔记规模下的快速检索与导航。
- 协作者共享空间、评论、权限、版本回滚。
- 本地智能索引、关系发现、AI 辅助整理与检索。
- 企业级发布、审计、加密和运维闭环。

## 1.6 建议的需求分层

为避免产品范围失控，建议定义如下优先级体系。

### P0：当前必须做强的核心能力

- 桌面工作区稳定性与启动速度。
- 富文本编辑体验与自动保存可靠性。
- 文件夹/标签/收藏/回收站的基本信息架构。
- 认证与用户设置的可靠闭环。
- 桌面窗口、托盘和快捷键体验。
- 发布与质量门禁可持续。

### P1：从“好用”走向“领先”的增强能力

- 本地缓存、离线可读、断网保护。
- 全文搜索与命令面板统一搜索入口。
- 工作区虚拟化、10k+ 笔记性能优化。
- 版本历史、误删恢复、快照导出。
- 自动更新、发布渠道、签名和可信交付。

### P2：形成头部产品差异化的能力

- 多端同步与冲突解决。
- 团队空间、共享、权限、审计。
- 双向链接、引用、关系视图和知识发现。
- AI 辅助整理、摘要、问答、搜索增强。
- 插件/扩展机制。

### P3：平台化能力

- 企业级 SSO、DLP、细粒度权限和审计。
- 私有化部署和监管域适配。
- 管理后台、运营指标、租户治理。

## 1.7 非功能性需求标准

行业领先产品不只看功能，还要在非功能维度达标。

| 维度 | 目标标准 | 当前判断 |
|---|---|---|
| 可用性 | 主路径低学习成本、状态可见、恢复简单 | 良好 |
| 性能 | 启动快、搜索快、长文编辑稳定、海量笔记不退化 | 当前仍有明显提升空间 |
| 安全 | 会话保护、最小权限、可信发布、审计可追溯 | 基础具备，但未达领先 |
| 可维护性 | 高内聚、低耦合、边界明确、易测试 | 当前表现较好 |
| 可扩展性 | 支持增加本地索引、同步、协作、AI、插件 | 架构基础较好 |
| 可运营性 | 监控、告警、灰度、回滚、质量门禁 | 当前不足 |

## 1.8 产品需求评估标准

建议后续所有 PRD、架构评审和版本验收统一遵循以下标准：

| 标准项 | 达标要求 |
|---|---|
| 目标用户清晰 | 每个版本必须明确服务对象、核心场景、成功指标 |
| 范围可控 | 明确区分 P0/P1/P2，不允许“顺手扩需求” |
| 现状与目标分离 | 文档必须区分“已实现”“规划中”“目标架构” |
| 非功能要求显式化 | 每个大版本必须给出性能、安全、质量门禁 |
| 验收标准可量化 | 不是“更好”，而是明确 SLA/SLO、测试和发布门禁 |
| 与代码一致 | 每项已实现能力都能追溯到路由、页面、store、repository、desktop 或 workflow |

## 1.10 本文结论

SDKWork Canvas 当前最正确的战略，不是继续横向堆功能，而是先把下面四件事做成行业领先：

1. 桌面工作区效率与稳定性。
2. 本地优先和搜索能力。
3. 安全、测试、发布的工程硬标准。
4. 为同步、协作、AI 和插件预留清晰边界。

如果这四项做对，后续无论走个人高级笔记产品，还是团队知识平台，架构都不会推倒重来。
