# Step 02 依赖方向决议

## 1. 决议目标

冻结后续本地层、搜索、同步、观测和更新能力的代码落点与依赖方向，避免继续默认堆进 `canvas-canvas`。

## 2. 决议内容

### 2.1 共享壳与业务壳

- `@sdkwork/canvas-shell` 继续作为共享业务壳事实来源
- `@sdkwork/canvas-desktop` 继续只做平台壳，不承接业务规则
- Web/Desktop 继续共享 `AppRoot`

### 2.2 未来能力包归属

- `@sdkwork/canvas-local`：本地存储与草稿恢复
- `@sdkwork/canvas-search`：索引和统一查询
- `@sdkwork/canvas-sync`：同步队列与冲突恢复
- `@sdkwork/canvas-observability`：日志、指标、事件
- `@sdkwork/canvas-updater`：桌面更新与渠道治理

### 2.3 禁止事项

- 不允许把本地权威层实现直接塞回 `@sdkwork/canvas-canvas`
- 不允许把同步状态机实现塞回页面组件或壳层
- 不允许把桌面更新实现塞回 `canvas-desktop` 业务 UI 组件层
- 不允许让 `canvas-shell` 直接持有未来能力包的具体实现细节

### 2.4 推荐依赖方向

```text
canvas-shell
  -> auth / canvas / user
    -> core / commons / i18n / types

canvas-canvas
  -> canvas-local / canvas-search / canvas-sync (后续分步接入)

canvas-desktop
  -> canvas-updater / platform runtime

cross-cutting
  -> canvas-observability
```

### 2.5 当前限制

- 当前只冻结骨架与边界，不等于这些包已经可以被业务层直接稳定消费
- 真实接入顺序仍然必须遵守 `Step 03 -> 04 -> 05 -> 06 -> 07 -> 08 -> 09`

## 3. 结论

- 从本决议起，`canvas-canvas` 不再是未来能力的默认堆积区
- 后续所有新增实现，应优先写入对应未来能力包，再通过 service / adapter 边界接入业务域
