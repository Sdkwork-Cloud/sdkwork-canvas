# Step 04 - 文件夹变更协调收敛

- 日期：`2026-04-07`
- Step：`04-工作区边界收敛与数据访问抽象`
- 本轮主题：`folder delete / move immediate coordination`
- 当前结论：`文件夹删除与移动的即时状态协调已从 store 收敛到独立服务，但 Step 04 仍为 L3`

## 1. 本轮目标

在完成初始化链回退收口后，本轮继续处理 Step 04 的下一个真实缺口：

- 删除文件夹后，`selectedFolderId / expandedFolderIds` 的协调不应继续散落在 store。
- 移动文件夹后的合法性校验与“展开新父节点”规则，也不应继续由 store 内联决定。

目标不是增加新功能，而是把这段 mutation 级即时状态协调抽成可测试的独立服务边界。

## 2. 实际改动

### 2.1 新增文件夹变更协调服务

`packages/sdkwork-canvas-canvas/src/services/noteWorkspaceFolderMutationCoordinator.ts`

新增纯函数服务：

- `planDeletedFolderState()`
- `planMovedFolderState()`
- `INVALID_FOLDER_MOVE_MESSAGE`

该服务负责：

- 计算被删除文件夹的整棵子树
- 删除后清理失效的 `selectedFolderId`
- 删除后裁剪失效的 `expandedFolderIds`
- 移动前校验“不能移动到自身或子孙节点下”
- 移动后补齐新父节点展开态

### 2.2 store 接线收口

`packages/sdkwork-canvas-canvas/src/store/useNotesWorkspaceStore.ts`

- `deleteFolder()` 不再自行计算删除子树与展开态裁剪，改为消费 `planDeletedFolderState()`
- `moveFolder()` 不再自行实现 descendant 校验与展开新父节点逻辑，改为消费 `planMovedFolderState()`

### 2.3 合同门禁接入

新增：

- `sdkwork-canvas-pc-react/scripts/workspace-folder-mutation.contract.test.mjs`

并接入：

- `sdkwork-canvas-pc-react/package.json`
- `sdkwork-canvas-pc-react/scripts/package-scripts-contract.test.mjs`

## 3. 结果判断

本轮之后，可以确认以下事实：

- 文件夹删除/移动的关键即时状态协调已不再直接散落在 store。
- 文件夹 mutation 的关键边界已具备独立 Node 合同门禁。
- `test:workspace:contracts` 现在可以持续冻结这段规则，不会在后续 Step 中被悄悄回退。

3. repository 仍缺少 `read-through / replica / sync` 的策略级接口面。
