# 95. 架构能力闭环验收标准

## 1. 文档定位

本文件用于判定某个 Step、某个波次、或者整个 `docs/step/` 规划是否真的兑现了 `docs/架构/` 的目标能力，而不是只完成了局部代码改动。

“闭环”在本项目中的定义是：

- 能力有清晰设计
- 能力有明确落点
- 能力有测试证明
- 能力有结果验证
- 能力有文档回写
- 能力对下一阶段形成稳定输入

只满足其中一部分，不算闭环。

## 2. 通用闭环判定模型

每项架构能力必须同时满足以下六类标准：

| 闭环维度 | 标准 |
| --- | --- |
| 设计闭环 | 有边界定义、状态模型、风险与回滚 |
| 实现闭环 | 有稳定模块落点，避免散落实现 |
| 测试闭环 | 有与风险等级匹配的测试覆盖 |
| 验证闭环 | 有可重复验证的场景、命令和结果 |
| 文档闭环 | `docs/架构/`、`docs/step/`、`docs/review/` 同步 |
| 集成闭环 | 与其他能力拼接后不破坏整体系统假设 |

## 3. 能力级闭环验收标准

### 3.1 共享壳层与工程骨架

验收目标：

- Web/Desktop 共享单一 `AppRoot` 事实来源
- 新能力具备清晰包落点
- 页面组件不承载平台级基础设施

通过标准：

- 壳层职责矩阵清晰
- Provider 组合、路由边界、平台适配边界清晰
- 新增能力不需要默认塞进 `canvas-canvas`

必备证据：

- 包边界说明
- contract test
- Web/Desktop 启动冒烟
- 架构回写

失败信号：

- 两端壳层分叉
- 新能力仍在页面内部直接实现
- 后续 Step 无法判断该写入哪个包

### 3.2 会话安全与配置治理

验收目标：

- 敏感会话不再长期依赖不安全持久层
- 配置分层、来源和覆盖优先级清晰

通过标准：

- token/refresh token 等敏感项有安全存储方案
- 配置项分类清晰：构建期、运行期、用户偏好、本地缓存
- 失败恢复、登出清理、迁移兼容可验证

必备证据：

- 安全存储迁移文档
- 配置 contract test
- 安全回归记录

失败信号：

- 仍长期依赖 `localStorage` 持久化高敏会话
- 不同环境行为不可预测
- 桌面与 Web 安全策略两套分叉

### 3.3 工作区初始化与数据访问边界

验收目标：

- 初始化链路稳定、可观察、可拆解
- repository/store/facade 边界清晰

通过标准：

- 初始化流程具备明确阶段划分
- 首屏与增量加载策略可解释
- 数据访问不绕过边界层直接散落在页面

必备证据：

- 时序图
- 集成测试
- 首屏性能基准

失败信号：

- 页面组件直接发请求和拼装数据
- 初始化链无法定位慢点或失败点
- 一处改动会牵动多个页面逻辑

### 3.4 编辑器可靠性与自动保存

验收目标：

- 用户输入可被稳定持久化
- 保存失败可见、可恢复、可重试

通过标准：

- 编辑状态、保存状态、同步状态分层清晰
- 自动保存具备节流/合并/失败回放机制
- 切换笔记、断网、异常退出不轻易丢稿

必备证据：

- 编辑保存状态机
- 关键回归测试
- 崩溃恢复验证

失败信号：

- 保存逻辑散落在多个组件
- 保存失败无明确 UI 反馈
- 切换页面或崩溃容易丢稿

### 3.5 本地存储层与离线草稿

验收目标：

- 建立本地优先基础
- 离线草稿和恢复闭环成立

通过标准：

- 本地模型有权威源定义
- 有 schema/迁移策略
- 离线编辑和恢复路径可验证

必备证据：

- schema 文档
- 迁移测试
- 离线/崩溃恢复脚本

失败信号：

- 本地数据只是临时缓存，没有清晰权威定义
- 迁移不可控
- 离线后用户状态不可预期

### 3.6 搜索索引与统一检索

验收目标：

- 搜索不是 UI 功能，而是能力底座
- 命令面板、搜索框、快捷入口共享统一语义

通过标准：

- 索引模型、更新机制、查询接口一致
- 标题、正文、标签、视图范围可被统一检索
- 10k 笔记规模下核心查询仍稳定

建议性能指标：

- 本地 10k 笔记全文查询 `P95 < 150ms`
- 命令面板首次可交互检索 `P95 < 200ms`

必备证据：

- 搜索准确性用例
- 性能基准
- 索引构建/重建验证

失败信号：

- 搜索和命令面板结果不一致
- 搜索结果依赖页面现有数据而非统一索引
- 大规模数据下明显卡顿

### 3.7 同步队列与冲突恢复

验收目标：

- 同步具备最小状态机
- 用户能够感知和处理失败/冲突

通过标准：

- 同步状态、重试、回放、冲突分类明确
- 离线后恢复同步不会破坏本地草稿
- 至少具备“一期可用”的冲突恢复交互

必备证据：

- 同步状态机
- 冲突演练记录
- 离在线切换验证

失败信号：

- 同步失败只静默重试
- 冲突后数据覆盖不可追溯
- 本地与远端谁是最终结果不明确

### 3.8 桌面壳能力与可信发布链

验收目标：

- 桌面运行时稳定
- 发布链可审计、可回滚、可信任

通过标准：

- 桌面能力矩阵清晰
- 关键平台能力通过适配层暴露，不污染业务层
- Release dry-run 可执行，签名、校验和产物追溯可验证

必备证据：

- 桌面 contract test
- bundle 校验
- workflow dry-run 记录

失败信号：

- 桌面功能直接散落在业务模块
- 只会打包，不具备可信校验与回滚设计
- 发布失败时无法快速定位和处理

### 3.9 测试体系与质量门禁

验收目标：

- 让测试和门禁接管“正确性证明”责任

通过标准：

- 单测、集成、E2E、视觉、性能、安全等适用层形成矩阵
- 主路径不再依赖口头说明验证
- workflow 具备明确的失败阻断作用

必备证据：

- 测试矩阵
- CI 结果
- 审计记录

失败信号：

- 只有局部单测
- 关键路径没有 E2E 或集成验证
- 发布前仍只能靠人工逐项点击

### 3.10 性能与大规模工作区治理

验收目标：

- 面向 10k+ 笔记规模保持可用性
- 性能优化从“经验修补”变为“持续治理”

通过标准：

- 启动、列表、搜索、编辑、切换等核心路径有基准
- 热点明确，优化措施可解释
- 回归可自动发现

建议性能指标：

- 工作区首屏准备时间明显优于当前基线
- 列表滚动和切换无明显主线程阻塞
- 大文档编辑保持稳定输入反馈

必备证据：

- 性能画像
- 基准报告
- 回归脚本

失败信号：

- 优化没有前后对比数据
- 大规模场景下体验明显退化
- 每次新增能力都破坏既有性能

### 3.11 版本历史与恢复能力

验收目标：

- 用户能查看、比较并恢复历史版本

通过标准：

- revision 模型明确
- 保留策略、生成时机、恢复语义明确
- UI 中能够清晰呈现恢复后结果

必备证据：

- 历史恢复测试
- diff 正确性验证
- 回滚演练

失败信号：

- 历史仅能展示，不能恢复
- 恢复后状态不可预测
- 恢复会破坏同步或本地层一致性

### 3.12 发布就绪与持续迭代闭环

验收目标：

- 形成 go/no-go 判断能力
- 为下一轮迭代提供稳定输入

通过标准：

- 当前版本候选结论明确
- 已知风险、回滚、遗留项、下一轮 backlog 明确
- `docs/架构/` 与 `docs/step/` 反映真实状态

必备证据：

- 波次验收记录
- 发布候选清单
- 回写审计与 backlog

失败信号：

- 只有“基本完成”的口头结论
- 没有发布风险与回滚结论
- 文档与代码现实再次脱节

## 4. 闭环评分建议

建议每项能力按以下评分：

| 闭环等级 | 说明 |
| --- | --- |
| `L0` 未启动 | 没有明确设计和落点 |
| `L1` 已设计 | 有方案，但未落地 |
| `L2` 已实现 | 有代码，但测试或验证不足 |
| `L3` 已验证 | 设计、实现、测试、验证完成，但文档/集成不足 |
| `L4` 已闭环 | 设计、实现、测试、验证、文档、集成全部完成 |

只有达到 `L4`，才可在整体架构中宣称能力已兑现。

## 5. 判定模板

建议每个能力在阶段验收时补充一份判定：

```md
## 能力闭环判定

- 能力编号：
- 当前等级：`L0/L1/L2/L3/L4`
- 设计闭环：是/否
- 实现闭环：是/否
- 测试闭环：是/否
- 验证闭环：是/否
- 文档闭环：是/否
- 集成闭环：是/否
- 结论：
- 仍缺失项：
```

## 6. 结论

架构能力的价值，不在于文档里写了多少目标，而在于是否最终形成可验证、可交付、可持续演进的闭环。本文件用于把“闭环”从口号变成明确标准。
