# Step 04 对话框状态收敛评审

- 日期: 2026-04-07
- Step: 04
- 当前能力等级: L3
- 评审范围: `pendingDialog` 解析、对话框标题/描述/确认文案生成、workspace contract 聚合回归

## 1. 评审结论

本轮增量已将 `NotesWorkspacePage.tsx` 中与 `pendingDialog` 相关的展示态拼装逻辑抽离到独立服务 `noteWorkspaceDialogState.ts`。页面层不再直接负责解析删除笔记、删除文件夹、清空回收站三类对话框的标题、描述与确认文案，而是消费统一的 dialog state。

这次收敛的直接收益如下：

1. 页面减少了一段高分支密度的条件判断。
2. 对话框文案逻辑拥有独立 contract，可单独回归。
3. 缺失实体时的默认标题策略被显式固化，避免未来 UI 文案回退不一致。
4. 新 contract 已并入 `test:workspace:contracts`，回归链完整。

## 2. 设计与实现评审

### 2.1 设计合理性

- 优点: 将展示态构建从 React 页面移到纯函数服务，边界明确。
- 优点: 服务只依赖 `pendingDialog`、`canvas`、`trashedNotes`、`folders`、`t`，不依赖 DOM、Hook、Router 或 Store 写操作。
- 优点: 支持未来扩展更多 dialog 类型时，在单一位置追加分支与测试。

### 2.2 实现逻辑评审

- `clearTrash` 直接生成固定文案，逻辑清晰。
- `deleteNote` 会在 `canvas + trashedNotes` 中解析目标标题，缺失时回退到 `canvas.defaults.docTitle`。
- `deleteFolder` 会在 `folders` 中解析目标名称，缺失时回退到 `canvas.defaults.folderTitle`。
- 页面层仅消费 `open/title/description/confirmLabel`，不再保留文案构建细节。

### 2.3 架构质量评估

- 高内聚: 对话框展示态统一收敛到单一服务，达成。
- 低耦合: 页面不再耦合具体文案生成规则，达成。
- 易测试: 纯函数 + Node contract，达成。
- 易扩展: 后续新增“批量删除”“冲突覆盖确认”等 dialog 类型时可继续扩展，基本达成。

## 4. 评估标准与达成情况

| 评估项 | 标准 | 结果 |
| --- | --- | --- |
| 展示态边界 | 页面不得直接拼装 dialog 标题、描述、确认文案 | 达成 |

## 5. 当前缺口

Step 04 仍保持 `L3`，原因如下：

1. 页面仍负责 `pendingDialog` 的创建与 `confirmPendingDialog()` 运行时协调。
2. 自动保存生命周期装配、恢复笔记流程、创建笔记流程等仍在页面内。
3. 当前仅完成“展示态收敛”，尚未完成“对话框运行时协调收敛”。

## 6. 下一步建议

建议继续优先以下方向之一：

1. 抽离 dialog runtime coordinator，收拢 `open -> confirm -> execute -> close` 闭环。
2. 抽离 restore/create note flow，继续压缩页面中的异步流程控制。
3. 抽离 autosave runtime coordinator，进一步收拢页面生命周期装配。
