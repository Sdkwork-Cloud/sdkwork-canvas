# 02. 架构标准与总体设计

## 2.1 架构目标

SDKWork Canvas 的总体架构目标不是“把 React 页面跑起来”，而是建立一个满足以下要求的长期演进底座：

- 高内聚：领域逻辑、桌面逻辑、壳层逻辑和共享能力清晰收口。
- 低耦合：模块之间通过稳定接口协作，而非跨层直接依赖。
- 易扩展：后续可平滑加入本地索引、同步引擎、协作、AI、插件。
- 高性能：桌面启动、工作区加载、编辑保存、搜索和切换都稳定。
- 高安全：会话、配置、桌面能力、发布链和质量门禁可治理。
- 易维护：包边界清晰、测试可覆盖、流水线稳定、文档可持续更新。

---

## 2.2 架构设计原则

### 2.2.1 核心原则

1. **体验层与业务层分离**  
   路由壳、桌面壳、主题与语言属于体验编排层；笔记、用户、认证属于业务域层。

2. **共享能力下沉到可复用包**  
   通用 UI、类型、i18n、SDK 接入、状态基础能力应收敛到独立包，而不是散落在页面里。

3. **平台壳只暴露最小必要能力**  
   Tauri 只负责窗口、托盘、语言配置等桌面壳能力，不直接侵入业务域。

4. **领域逻辑优先于页面逻辑**  
   页面组件主要编排状态和交互，真正的规则应沉淀到 `repository / services / store selectors`。

5. **当前现状与目标架构并行治理**  
   允许当前先使用远端笔记服务，但必须为本地优先和同步架构预留演进接口。

### 2.2.2 质量属性原则

| 质量属性 | 原则 |
|---|---|
| 性能 | 面向大规模数据提前设计分页、索引、虚拟化、缓存和异步任务 |
| 安全 | 默认最小权限、最小暴露、最小持久化 |
| 可观测 | 每条关键链路都应有日志、指标和错误归因 |
| 可回滚 | 配置、发布、数据同步都必须支持回退策略 |
| 可测试 | 页面、服务、仓储、脚本、桌面桥都要有分层测试 |

## 2.3 当前总体架构

当前架构可以抽象为五层。

### 2.3.1 当前入口拓扑

当前不是单一桌面入口，而是共享业务壳的双入口拓扑：

```text
Web 入口
  src/main.tsx
  -> ensureI18n
  -> App
  -> AppRoot

Desktop 入口
  packages/sdkwork-canvas-pc-desktop/src/main.tsx
  -> createDesktopApp
  -> startupAppearance snapshot
  -> waitForTauriRuntime(<= 600ms)
  -> hasNativeRuntime ? configureDesktopPlatformBridge() : skip
  -> ensureI18n
  -> DesktopBootstrapApp
  -> AppRoot
```

这一设计的关键收益是：

- Web 与 Desktop 共享同一业务壳、路由和业务域，不形成双套前端。
- Desktop 只叠加启动屏、桌面桥、托盘路由桥和桌面语义属性，不侵入领域实现。
- 如果 Desktop 入口在限定时间内未探测到 Tauri runtime，仍会继续渲染共享业务壳，并通过 bridge fallback 保持安全可运行。

### 2.3.2 当前层次结构

```text
桌面宿主层
  Tauri 2 + Rust
  - 窗口生命周期
  - 托盘菜单与导航
  - 语言配置写入
  - 运行时信息投影
  - 启动外观快照与桌面桥接注入

应用壳层
  canvas-pc-shell
  - 路由
  - 主布局
  - AppProviders
  - ShellLayout / AppHeader
  - 主题、语言、通知

业务域层
  canvas-pc-auth / canvas-pc-canvas / canvas-pc-user
  - 认证
  - 笔记工作区
  - 账号设置

共享基础层
  canvas-pc-core / canvas-pc-commons / canvas-pc-i18n / canvas-pc-types
  - app-sdk 客户端
  - 用户服务、会话编排与运行时配置治理
  - 偏好存储
  - 通用组件
  - 国际化
  - 类型模型

远端能力层
  @sdkwork/app-sdk
  - note
  - filesystem
  - user
  - auth
```

### 2.3.3 当前架构优点

- 包拆分清晰，按业务域和共享能力划分。
- 双入口共享 `AppRoot`，避免 Web 与 Desktop 业务逻辑分叉。
- 桌面宿主层与业务层边界清晰，Tauri 没有侵入业务组件。
- 认证、用户设置和笔记工作区具备独立边界。
- 通过 `@sdkwork/app-sdk` 统一接入远端能力，避免页面直接拼 HTTP。
- 具备共享主题、国际化、通用组件和统一路由壳。
- 启动外观快照、认证壳/业务壳分流和桌面窗口控件等“桌面工作台语义”已经沉淀在壳层，而不是散落在页面内部。

### 2.3.4 当前架构短板

- 笔记主数据仍高度依赖远端服务，不具备本地优先能力。
- 初始化和搜索仍偏“客户端扫描 + 客户端聚合”模式。
- 偏好和界面状态仍部分落在 `localStorage`；会话已迁移到受控会话层，但距离 OS 级安全存储仍有升级空间。
- 缺少专门的同步层、索引层、审计层和可观测层。
- 缺少对大规模数据和多端协作的架构预留。
- 当前主文档此前更偏 Desktop 叙事，容易弱化 Web 入口、Web bundle 和桥接 fallback 这些真实交付拓扑事实。

## 2.4 目标总体架构

为了达到行业领先水平，建议将总体架构升级为六层。

### 2.4.1 目标层次结构

```text
L1 体验与交互层
  - 路由壳
  - 窗口与工作区布局
  - 编辑器、检视器、命令面板
  - 主题、国际化、可访问性

L2 应用编排层
  - 应用状态
  - 会话编排
  - 导航状态
  - 通知、错误恢复
  - Feature Flags

L3 领域服务层
  - 笔记领域服务
  - 文件夹/空间领域服务
  - 用户与设置服务
  - 搜索服务
  - 协作服务

L4 本地能力层
  - 本地数据库
  - 本地全文索引
  - 附件缓存
  - 同步队列
  - 本地加密与密钥管理

L5 同步与平台层
  - 增量同步
  - 冲突合并
  - 权限/共享
  - 版本历史
  - 审计与事件流

L6 观测与交付层
  - 日志与指标
  - 崩溃上报
  - 配置中心
  - 发布渠道
  - 质量门禁
```

### 2.4.2 目标架构关键收益

- 支撑离线编辑、本地搜索、断网恢复。
- 支撑大规模笔记数据的本地索引与快速导航。
- 支撑多端同步、冲突处理与版本历史。
- 支撑企业级安全、发布和观测闭环。
- 支撑 AI 能力和插件能力在稳定边界上落地。

## 2.5 质量属性目标

建议统一采用如下架构级质量目标。

| 维度 | 目标值 | 说明 |
|---|---|---|
| 冷启动时间 | `P95 < 2.5s` | 桌面启动到主工作区可交互 |
| 工作区首屏可用 | `P95 < 1.8s` | 已登录情况下进入最近工作区 |
| 本地搜索延迟 | `P95 < 150ms` | 10k 笔记规模下 |
| 自动保存确认 | 本地 `P95 < 300ms`，同步 `P95 < 1s` | 需要本地和远端双层确认 |
| 崩溃率 | `Crash-free sessions > 99.8%` | 版本级质量目标 |
| 数据恢复 | 单设备异常退出不丢最近 30 秒编辑 | 需要本地日志与恢复机制 |
| 发布回滚 | 30 分钟内可回退到前一稳定版本 | 需渠道与制品治理支持 |

## 2.6 架构治理标准

### 2.6.1 模块治理标准

| 标准项 | 要求 |
|---|---|
| 依赖方向 | 页面依赖服务，服务依赖仓储/适配器，禁止反向依赖 |
| 包职责 | 一个包只服务一个主要领域或一种共享能力 |
| 公开面 | 每个包通过 `index.ts` 统一收口导出 |
| 状态边界 | UI 层状态和领域状态分离，不混用 |
| 平台边界 | Tauri API 只允许在 `canvas-pc-desktop` 内部封装暴露 |

### 2.6.2 架构决策标准

| 标准项 | 要求 |
|---|---|
| ADR | 对关键技术决策必须形成 ADR |
| 向后兼容 | 外部接口和共享包导出必须遵守兼容策略 |
| 可观测性 | 新增关键流程必须定义日志、错误和指标 |
| 退场机制 | 新引入能力必须定义禁用、回退和迁移方案 |
| 安全审查 | 任何涉及会话、文件、执行权限的变更必须过安全审查 |

| 项目 | 当前判断 | 评分（10 分） |
|---|---|---:|
| 分层合理性 | 良好 | 8.7 |

## 2.8 总体设计评估标准

架构评审建议采用下面的统一标准：

| 评估项 | L3 合格 | L4 优秀 | L5 行业领先 |
|---|---|---|---|
| 分层 | 能分清 UI/业务/基础层 | 有稳定边界与适配器 | 支持多端、多团队长期演进 |
| 扩展性 | 能增加新页面/新模块 | 能增加新能力不重构旧架构 | 可平滑引入同步、AI、插件、协作 |
| 性能架构 | 主路径可用 | 有缓存、索引、虚拟化策略 | 大规模数据下仍稳定 |
| 安全架构 | 有基础会话保护 | 有最小权限与安全门禁 | 有零信任、密钥治理、可审计发布 |
| 运维架构 | 可发布 | 有质量门禁和回滚 | 有全链路观测和渠道治理 |

## 2.9 本文结论

SDKWork Canvas 当前的总体设计方向是正确的：

- 选择了适合桌面产品的 `React + Tauri + Rust` 组合。
- 选择了“共享 AppRoot + Web/Desktop 双入口 + 桌面增强壳”的启动拓扑。
- 选择了按业务域拆包的模块化架构。
- 选择了以共享 SDK 为中心的远端能力接入方式。

但如果目标是“行业最领先的笔记软件”，下一阶段必须把总体设计从“云端驱动桌面应用”升级为“本地增强、同步驱动、观测完备的知识平台客户端”。
