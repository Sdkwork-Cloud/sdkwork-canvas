# 06-业务流程-应用接口与集成设计-页面洞察区组件收敛补充-2026-04-07

## 1. 背景

在页面展示模型和页面 chrome 规则已经下沉后，`NotesWorkspacePage.tsx` 仍残留一块纯视图装配胶水：洞察区的指标卡渲染、焦点卡 detail 渲染，以及页面本地 `WorkspaceMetricCard` 辅助组件。

这部分逻辑不再承担业务编排，但仍会持续抬高页面容器厚度，使页面难以进一步稳定为薄适配层。

## 2. 设计决策

本轮新增 `NotesWorkspaceInsightsPanel.tsx`，将以下职责从页面容器中剥离：

1. 指标卡列表渲染
2. 焦点卡主区块渲染
3. 焦点卡 badges 与 details 渲染
4. 本地 `WorkspaceMetricCard` 渲染辅助

### 输入

- `focusTitle`
- `pagePresentation`

### 依赖

- `resolveNotesWorkspaceChromeIcon()`

### 页面层保留职责

- 负责传入 `focusTitle`
- 负责传入 `pagePresentation`
- 负责整体页面 section 结构、状态装配和命令绑定

## 3. 设计价值

### 高内聚

洞察区的所有渲染责任现在集中在单一组件中，页面不再同时承担“页面壳层 + 洞察区细节组件”双重责任。

### 低耦合

洞察区组件仅消费展示模型和 icon 解析服务，不需要了解 store、runtime 或路由逻辑。

### 易维护

后续继续迭代洞察区布局、响应式变化或视觉细节时，可直接在该组件内演进，而不会重新膨胀页面容器。

## 4. 评估标准

本轮页面洞察区组件收敛是否达标，使用以下标准判断：

1. `NotesWorkspacePage.tsx` 不得继续定义 `WorkspaceMetricCard` 本地辅助组件。
2. `pagePresentation.metricCards.map(...)` 与 `pagePresentation.focusCard.details.map(...)` 不得继续停留在页面本体。
3. 必须存在独立的 `NotesWorkspaceInsightsPanel.tsx` 承接洞察区渲染。
4. 必须建立显式页面容器边界 contract，冻结“洞察区必须退出页面层”的事实。
5. contract 必须进入 `test:workspace:contracts` 并被 `package-scripts-contract` 冻结。

## 5. 当前残留与后续方向

本轮之后，Step 04 的页面侧主阻塞进一步缩小为：

1. header action 到最终按钮/链接节点的页面级绑定。
2. command palette descriptor 到 `onSelect` 闭包的页面级绑定。

因此，本轮属于页面容器继续瘦身，但尚不足以支撑 Step 04 进入 `L4`。
