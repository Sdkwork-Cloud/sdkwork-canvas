# 对话框状态补充设计

- 日期: 2026-04-07
- 所属 Step: 04
- 主题: 工作区对话框展示态边界收敛

## 1. 背景

在笔记工作区中，删除笔记、删除文件夹、清空回收站均通过统一的 `Dialog` 组件呈现，但原实现将以下职责同时放在页面中：

1. 根据 `pendingDialog` 判断当前对话框类型。
2. 解析目标 note/folder 实体。
3. 生成标题、描述和确认按钮文案。
4. 处理缺失实体时的默认回退文本。

这类逻辑虽然不涉及写操作，但属于明显的“页面展示态决策”，如果持续留在页面中，会让页面承担过多 UI 语义拼装责任。

## 2. 当前设计

### 2.1 服务边界

文件: `packages/sdkwork-canvas-canvas/src/services/noteWorkspaceDialogState.ts`

职责:

- 将 `pendingDialog` 解析为 `Dialog` 可直接消费的 `open/title/description/confirmLabel`
- 统一处理缺失 note/folder 的回退文案
- 作为页面与 i18n 文案之间的中间边界

### 2.2 输入输出

输入:

- `pendingDialog`
- `canvas`
- `trashedNotes`
- `folders`
- `t`

输出:

- `open`
- `title`
- `description`
- `confirmLabel`

### 2.3 页面职责

页面仅保留两类职责：

1. 持有 `pendingDialog` 状态。
2. 消费 `buildNotesWorkspaceDialogState()` 的输出渲染 `Dialog`。

## 3. 架构价值

### 3.1 高内聚

所有与 dialog 展示态相关的判断都集中在单一 service，不再散落在页面局部变量中。

### 3.2 低耦合

页面不需要知道“不同 dialog 类型如何拼装文案”，只需要渲染结果。

### 3.3 易维护

当 i18n key、默认回退策略或 dialog 类型新增时，只需修改服务与契约，而非修改页面多个分支。

### 3.4 易扩展

未来可扩展：

- 批量删除确认
- 冲突覆盖确认
- 同步失败重试确认
- 本地缓存清理确认

## 4. 评估标准

| 维度 | 标准 | 当前状态 |
| --- | --- | --- |
| 语义收敛 | dialog 展示态必须由单一服务统一生成 | 达成 |
| 回退稳定性 | 缺失实体时必须有稳定默认文案 | 达成 |
| 页面简洁度 | 页面不再保留多段标题/描述拼装分支 | 达成 |
| 契约化 | 关键对话框文案决策必须可独立回归 | 达成 |
| 扩展性 | 新增 dialog 类型不应要求大幅修改页面 | 基本达成 |

## 5. 对标行业先进标准

行业领先的笔记应用在 destructive action 相关的交互上，通常要求：

1. 文案与目标实体绑定关系清晰。
2. 缺失数据场景仍具备稳定退化行为。
3. 新交互类型可以快速扩展，不破坏现有页面结构。

当前实现已完成展示态层面的收敛，但要达到更高成熟度，还需要继续把确认执行流程与副作用协调从页面中抽离出来。
