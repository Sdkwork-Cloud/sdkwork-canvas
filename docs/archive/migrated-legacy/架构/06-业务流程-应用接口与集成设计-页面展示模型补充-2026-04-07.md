# 06-业务流程-应用接口与集成设计-页面展示模型补充-2026-04-07

## 1. 背景

在 Step 04 已完成 create note、hotkey、sidebar resize、page command runtime 等页面运行时收敛之后，`NotesWorkspacePage.tsx` 仍残留一类纯展示胶水：

- 快捷键文案依赖 `navigator.platform` 的 `Cmd/Ctrl` 分支
- 统计卡片的 label/value/icon 装配
- 焦点卡片的标题、描述、状态 badge、细节卡片装配

这类逻辑不属于 React 生命周期，也不属于 store/runtime 业务编排，更适合被定义为纯展示模型服务。

## 2. 设计决策

本轮新增 `noteWorkspacePagePresentationModel.ts`，并定义以下边界：

### 输入

- `activeView`
- `activeNote`

- `activeTaskProgress`
- `activeOutline`
- `activeWordCount`

### 输出

- `focusCard`

### 页面层保留职责

- 将 icon key 映射为 Lucide 组件
- 绑定按钮点击与 runtime/store action
- 负责 JSX 结构与样式落点

## 3. 设计价值

### 高内聚

页面展示模型把“仅与展示有关、但不该散落在 JSX 内”的计算集中到单一服务，避免页面既做视图，又做展示规则组装。

### 低耦合

服务本身不依赖 React，不依赖 DOM，不依赖 store，仅消费输入数据并返回结构化模型，适合 Node contract 单独验证。

### 易扩展

后续若需要继续抽离：

- header action descriptor
- command palette icon strategy
- focus card layout variant

都可以沿同一模型层继续推进，而不必把逻辑重新塞回页面。

## 4. 评估标准

本轮页面展示模型收敛是否达标，使用以下标准判断：

1. 页面展示模型必须是纯函数，不能依赖 React hook、window、document。
2. `Cmd/Ctrl` 决策必须通过统一函数输出，页面不得重复判断平台。
3. badge/status/detail 的展示规则必须可被 Node contract 独立验证。
4. 页面组件不得再内联生成快捷键字符串、发布状态 class 与焦点卡片 badge/detail 数据。
5. 新增 contract 必须接入 `test:workspace:contracts`，并由 `package-scripts-contract` 冻结脚本链。

## 5. 当前残留与后续方向

本轮之后，页面展示胶水的残留进一步收敛为：

1. 命令面板与部分展示卡片的 icon map 仍在页面层。
2. header action button 的 descriptor 尚未模型化。
3. Step 04 之外，repository 面向未来同步能力的策略接缝仍需继续设计。

因此，本轮属于 Step 04 的增量完善，而不是 Step 04 的关闭条件。
