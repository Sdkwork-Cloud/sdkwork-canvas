# Step 02 - 共享壳层与工程骨架收敛

## 1. 目标与范围

本 step 用于把当前已经存在的多包结构进一步收敛成“可承载后续本地能力、搜索、同步、版本、观测和 updater”的稳定骨架。

### 1.1 执行输入

- Step 01 差距矩阵
- `docs/架构/02`、`03`、`04`、`06`
- 当前 `sdkwork-canvas-shell / canvas-core / canvas-canvas / canvas-desktop`

### 1.2 本步非目标

- 不在本 step 内直接实现本地数据库、全文索引、同步队列
- 不在本 step 内实现版本历史或自动更新

### 1.3 最小输出

- 共享壳层职责冻结
- 新能力包或目录的落点设计和初始骨架
- 导出面与依赖方向收敛

## 2. 架构对齐

- `docs/架构/02-架构标准与总体设计.md`
- `docs/架构/03-模块规划与边界.md`
- `docs/架构/04-技术选型与可扩展策略.md`

## 3. 当前现状与问题

当前包边界已较健康，但后续能力缺少明确落点：

- 新能力若不先定骨架，极易继续堆进 `canvas-canvas`
- Web / Desktop 共享壳已存在，但缺少对未来 `canvas-local / canvas-search / canvas-sync / canvas-observability / canvas-updater` 的工程收口

## 4. 设计

### 4.1 骨架目标

建议形成以下能力落点：

- `canvas-local`：本地存储、草稿恢复、未来 SQLite 适配
- `canvas-search`：本地索引、搜索 API、命令面板候选源
- `canvas-sync`：同步队列、游标、冲突状态、重试控制
- `canvas-observability`：日志、指标、事件封装
- `canvas-updater`：桌面更新、渠道、版本治理

### 4.2 依赖原则

- Web 与 Desktop 继续共享 `AppRoot`
- `canvas-desktop` 只做平台壳，不承载业务域规则
- 页面不直接依赖未来本地实现细节，统一通过 service / adapter 边界

## 5. 实施落地规划

1. 冻结未来能力包和目录命名
2. 明确各包 `index.ts` 导出面
3. 更新 workspace 边界约束与 contract test
4. 清理壳层与业务层跨包越界依赖

## 6. 测试计划

- `pnpm test:workspace:contracts`
- `pnpm typecheck`
- 壳层、桌面壳、包导出面相关单测与 contract test

## 7. 结果验证

完成后必须满足：

- 后续新能力已有稳定代码落点
- Web 与 Desktop 不会分叉两套业务壳
- `canvas-canvas` 不再成为未来所有能力的默认堆积区

## 8. 检查点

- `CP02-1`：未来能力包与目录落点冻结
- `CP02-2`：依赖方向和导出面冻结
- `CP02-3`：workspace 边界测试通过
- `CP02-4`：架构文档已回写

### 8.1 推荐 review 产物

- `docs/review/step-02-骨架收敛审计-YYYY-MM-DD.md`
- `docs/review/step-02-依赖方向决议-YYYY-MM-DD.md`

### 8.2 推荐并行车道

- `02-A`：共享壳和 provider 边界收敛
- `02-B`：新能力包骨架与导出面
- `02-C`：contract test 与文档回写

### 8.3 架构能力闭环判定

只有当新增能力不再需要默认落到 `canvas-canvas`，且 Web/Desktop 共享壳仍保持单一事实来源时，本 step 才算闭环。

### 8.4 快速并行执行建议

- 先冻结包边界，再允许后续 step 在各自包内并行
- 所有包名、职责和 owner 一次性定清，不要边做边改名

### 8.5 完成后必须回写的架构文档

- `docs/架构/03-模块规划与边界.md`
- `docs/架构/04-技术选型与可扩展策略.md`
- `docs/架构/09-实施计划.md`

### 8.6 本 Step 完成后必须兑现的架构能力

- Web/Desktop 共享壳层、Provider 组合和路由归属形成单一事实来源
- `canvas-local / canvas-search / canvas-sync / canvas-observability / canvas-updater` 等未来能力具备明确落点
- 导出面和依赖方向经过收敛，后续能力扩展不再默认堆进 `canvas-canvas`

### 8.7 最快完成的并行执行顺序

1. 先冻结能力包命名、目录落点和依赖方向
2. 再并行推进共享壳层收敛与新能力包骨架
3. 最后统一做导出面、contract test 和文档回写
4. 页面层调整必须留到边界冻结后再集中收口

## 9. 风险与回滚

### 9.1 风险

- 只建空包不收敛导出和依赖，后续仍会返工

### 9.2 回滚

- 包骨架可先以最小兼容实现存在，但导出面变更必须保留兼容 facade

## 10. 完成定义

- 共享壳边界冻结
- 新能力包落点冻结
- 边界测试通过

## 11. 下一步准入条件

进入 step 03 前必须确认：

- 会话安全与配置治理的代码落点已明确
- 未来本地能力不会直接侵入壳层
