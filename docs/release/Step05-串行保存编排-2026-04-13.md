# Step05-串行保存编排

- 日期：`2026-04-13`
- 版本：`0.1.0 / Unreleased`
- 阶段：`Step 05 / Wave-B / 第二十九轮推进`
- 范围：`保存链可靠性升级`

## 1. 变更摘要

本轮把活跃笔记保存请求正式收敛到串行 `save queue`，同时补齐高风险动作在保存进行中的等待语义，解决了“已有共享 flush 入口，但真实保存执行仍会并发交叠”的结构问题。

## 2. 已交付内容

1. `noteWorkspaceSaveQueue.ts` 新增独立保存编排服务，统一承接串行执行、replay 合并与等待语义。
2. `resolveNotesWorkspaceSaveCompletion()` 新增保存完成 helper，旧响应不再覆盖更新草稿。
3. `useNotesWorkspaceStore.ts` 改为在已有 active request 时区分：
   - `dirty / error -> requestReplay()`
   - 其它状态 -> `waitForActiveRequest()`
4. 高风险动作前的 `persistUnsavedActiveNoteIfNeeded()` 现在显式执行：
   - `dirty / error -> flush`
   - `saving / retrying -> wait`
5. `workspace-save-queue.contract.test.mjs` 与升级后的 `workspace-high-risk-flush-boundary.contract.test.mjs` 已接入主门禁。

## 3. 用户与架构收益

1. 连续编辑触发多次保存时，不再轻易出现并发写请求互相覆盖。
2. 用户在保存过程中切换笔记或执行危险动作时，会先等当前请求完成，而不是直接放行。
3. 旧保存响应返回时，不会把更新的本地草稿错误回退到更早快照。

## 5. 风险与回滚评估

### 5.1 已控制风险

1. 新 contract 已进入 `test:workspace:contracts`，保存编排或高风险刷盘边界一旦回退会直接暴露。
2. replay 只合并一次后续保存，不会在失败后继续无限回放。
3. store 只在当前草稿未发生新编辑时才更新 `activeNote`，降低隐性丢稿风险。

### 5.2 尚存风险

1. 自动退避、重试上限和保存观测仍未形成稳定方案。
2. 页面关闭、异常退出与崩溃恢复证据仍需继续补齐。
3. 当前仍主要依赖 contract + typecheck 作为验证主力，Vitest/E2E 受环境限制未形成同等证据。
