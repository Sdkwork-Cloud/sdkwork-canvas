# 06-业务流程-应用接口与集成设计-本地恢复入口补充

- 日期：`2026-04-13`
- 所属 Step：`Step 06`
- 波次：`Wave-B`
- 主题：`本地恢复草稿读侧接入与页面恢复入口`

## 1. 背景

在 `Step 05` 已经完成“退出恢复检查点主链”之后，`Step 06` 的第一条真实消费边界必须建立在已有本地恢复检查点之上，而不是重新定义另一套本地草稿写链。

本轮目标不是引入完整本地优先，而是把以下事实正式冻结：

1. `NotesLocalStore.loadWorkspace().drafts` 是当前唯一允许消费的恢复输入源。
2. 恢复动作仍以远端工作区为权威源，只把本地草稿当作可回放副本。
3. 恢复回放必须重新进入当前 `save queue + save feedback + retry policy + exit recovery capture` 主链。

## 2. 冻结后的恢复读侧模型

### 2.1 恢复候选筛选规则

恢复候选只允许满足以下条件：

1. `noteId` 必须仍然存在于当前 workspace 的 live note 列表。
2. `trashedNotes` 中的 note 不进入恢复面。
3. 当前工作区中不存在的 missing note 不进入恢复面。

这意味着当前恢复入口只负责“仍属于当前工作区、但本地仍保留未回放草稿”的场景，不提前承诺跨删除、跨冲突副本的复杂恢复语义。

### 2.2 Store 状态边界

`useNotesWorkspaceStore` 新增两类只读恢复状态：

1. `recoveredDrafts`
   - 当前工作区内所有可见恢复候选
2. `activeRecoveredDraft`
   - 当前 `activeNoteId` 对应的恢复候选

同时新增两个动作：

1. `restoreRecoveredDraft(noteId)`
   - 把本地恢复草稿字段回放到当前 `activeNote`
   - 状态重新置为 `dirty`
   - 重新复用 `draft-change capture`
2. `dismissRecoveredDraft(noteId)`
   - 清理本地恢复检查点
   - 从当前会话恢复面移除对应候选

## 3. 页面恢复入口规则

页面恢复入口现在分为两种：

1. 当前选中的笔记就是恢复候选
   - CTA：`恢复草稿 / 放弃草稿`
2. 恢复候选属于其它笔记
   - CTA：`打开笔记 / 放弃草稿`

这样可以保证：

1. 用户总能看到“最近可恢复状态”。
2. 恢复动作发生前，仍由现有 `selectNote()` 和 active note 装载流程托管上下文切换。
3. 页面层不直接操作 `localStore`，只消费 store 的恢复状态与动作。

## 4. 与 Step 05 主链的集成约束

### 4.1 允许的集成方式

本轮只允许以下方式接入：

1. 初始化时读取 `NotesLocalStore.loadWorkspace().drafts`
2. 恢复时回放到 `activeNote`
3. 恢复后继续走既有保存链

### 4.2 禁止的做法

以下做法当前仍然禁止：

1. 绕过 `persistActiveNote()` 直接新增第二条本地保存链
2. 页面层直接读写 `localStore`
3. 在恢复入口中把本地副本声明为新的权威源
4. 在 `Step 06` 一期提前实现冲突副本、多版本恢复或完全离线写入

## 5. 本轮后得到的架构事实

1. 本地恢复面已经从“只写不读”升级为“可读、可见、可回放”。
2. 远端仍是权威源，本地恢复只承担副本和回放职责。
3. `Step 07 / Step 08` 后续要接搜索索引和同步状态机时，仍应建立在当前本地恢复模型之上继续扩展，而不是推翻当前 store/page 接口。

## 6. 仍未闭环的架构缺口

1. `canvas / folders` 的稳定本地副本尚未建立。
2. schema version / migration 仍未冻结。
3. 搜索与同步尚未拿到统一的本地快照接口。
4. 删除与修改冲突当前仍只保留“后续需要冲突副本”的架构要求，还未在本轮实现。
