# 反复执行 Step 指令

适用仓库：`apps/sdkwork-canvas`

使用方式：

- 每一轮执行时，将下方整段 Prompt 原样输入。
- 每次输入后，执行者都必须基于仓库当前真实状态继续推进，而不是重新空谈计划。
- 本 Prompt 适用于持续执行 `docs/step/`、对齐 `docs/架构/`、补齐 `docs/review/`、维护 `docs/release/`，直到整个笔记应用达到可商业化交付标准。

---

## 可重复输入 Prompt

```md
你现在负责持续推进 `apps/sdkwork-canvas`。你的目标不是只完成一个局部任务，而是基于当前仓库真实状态，将 `docs/step/` 中定义的全部能力完整实现、验证、收口，并持续打磨到可商业化交付的专业笔记应用水平。

### 一、强制阅读与权威来源

开始执行前，必须先读取并以其为最高优先级事实来源：

- `docs/step/README.md`
- `docs/step/00-13` 全部 Step 文档
- `docs/step/90-99` 全部治理与验收文档
- `docs/架构/README.md`
- `docs/架构/01-10` 全部架构文档
- `docs/review/` 下最新执行卡、审计、复盘、阶段验收、架构兑现文档
- `docs/release/` 下现有 changelog、版本说明、发布说明
- 当前源码、测试、脚本、构建配置、CI/CD、Desktop runtime、安装与发布配置、Git 状态

如果文档、代码、测试、脚本之间存在冲突，必须先回到源码事实和最新治理文档判断，再决定修正文档还是继续实施，不允许带着冲突继续推进。

### 二、总目标

本项目的总目标是：

- 按 `docs/架构/` 与 `docs/step/` 的标准，把当前应用演进为行业领先的专业笔记应用。
- 对标优秀笔记类软件的领先能力，包括但不限于：Notion、Obsidian、Apple Notes、OneNote、Craft。
- 在标准、设计、功能、流程、实现逻辑、技术架构、技术选型、模块边界、组件设计、性能、安全、测试、安装、部署、发布、可维护性、可扩展性方面达到商业化交付水平。
- 确保整体设计始终满足：高内聚、低耦合、易扩展、高性能、高安全、易维护、易测试、易发布、易运维。
- 将“完美”定义为可审计、可验证、可发布、可回滚、可持续演进的工程完备状态，而不是主观感觉上的“差不多完成”。
- 在所有关键维度上至少达到“达标”，在核心竞争力维度上尽可能逼近“领先”，并将无法在本轮解决的事项显式沉淀为 backlog，而不是隐藏为未说明风险。

你必须始终围绕“真实现状 -> 差距识别 -> 设计对齐 -> 实施落地 -> 测试验证 -> 文档回写 -> 发布记录 -> 结果评估 -> 下一轮迭代”这个闭环推进。

### 三、绝对执行原则

- 不要停留在分析、建议、计划、口头总结。
- 不要重复已完成且已有证据闭环的内容。
- 不要把局部代码提交当作 Step 完成。
- 不要跳过测试、验证、`docs/review/`、`docs/架构/` 回写、`docs/release/` changelog 更新。
- 不要在架构边界不清、协议未定、共享状态机冲突、数据模型冲突时继续堆功能。
- 任何“已完成”的判断，都必须拿出代码、测试、验证、文档四类证据。
- 任何“商业化可交付”的结论，都必须包含发布就绪、风险清单、回滚策略、遗留项和下一轮计划。

### 四、防自我欺骗与自我纠偏原则

- 不要默认信任上一轮的“已完成”结论；每轮都必须重新检查关键证据是否真实存在、是否仍然有效。
- 如果证据不足、测试未跑、验证无法复现、文档未回写，则默认判定为“未完成”，而不是“基本完成”。
- 如果同一个问题连续两轮以上反复出现，不能继续按原策略蛮干，必须先做根因分析，再切换方案。
- 如果本轮引入了新的架构漂移、性能回退、安全回退、测试回退或发布回退，必须优先止损并修复回退，再继续扩展能力。
- 如果一个 Step 的功能表面可用，但 `91` 评分不达标、`95` 未达到 `L4`、或 `97` 未完成回写，则一律视为假完成。
- 如果“追求完美”会明显破坏当前边界稳定性或引入过度设计，必须回到商业化目标，优先选择可验证、可维护、可演进的最优方案，而不是复杂方案。

### 五、每轮开始必须执行的判断

每次接收到这段 Prompt 后，必须先做以下判断：

1. 扫描当前仓库真实状态，判断当前处于哪个 Step、哪个波次、哪个能力等级。
2. 严格按 `docs/step/95-架构能力闭环验收标准.md` 判断当前 Step 的闭环等级 `L0/L1/L2/L3/L4`。
3. 判断当前 Step 是否已满足以下六类闭环：
   - 设计闭环
   - 实现闭环
   - 测试闭环
   - 验证闭环
   - 文档闭环
   - 集成闭环
4. 判断当前波次是否已具备执行 `docs/step/93` 波次总验收的条件。
5. 判断上轮变更是否已经同步写入：
   - `docs/review/`
   - `docs/架构/`
   - `docs/release/`
6. 判断本轮哪些事项可并行，哪些必须串行，严格依据：
   - `docs/step/94`
   - `docs/step/96`
   - `docs/step/98`
   - `docs/step/99`

如果当前 Step 还未达到 `L4`，必须优先补齐当前 Step，不允许跳到下一 Step。

### 六、Step 推进规则

- 若当前 Step 未达到 `L4`：优先补齐当前 Step 的缺口，直到形成闭环。
- 若当前 Step 已达到 `L4`：自动进入下一个 Step，严格按照 `00 -> 01 -> 02 -> 03 -> 04 -> 05 -> 06 -> 07 -> 08 -> 09 -> 10 -> 11 -> 12 -> 13` 推进。
- 若某个 Step 因外部依赖、第三方能力、环境限制、跨团队决策而阻塞：
  - 明确记录阻塞项、影响范围、最小风险规避方案、恢复条件。
  - 不允许因为一个阻塞点让整个仓库进入停滞。
  - 必须继续推进其他不冲突的 Step 子任务或质量补齐项。
- 若 `00-13` 全部达到 `L4`：
  - 不要停止。
  - 自动进入持续优化模式。
  - 持续从性能、安全、体验、可靠性、发布能力、安装升级、兼容性、可观测性、恢复能力、代码整洁度、模块边界、测试覆盖等方面继续迭代。

### 七、并行 / 串行执行规则

先识别共享写入面、共享契约、共享状态机和主链路阻塞项，再决定是否并行。

必须串行执行的工作包括：

- 架构边界重定义
- 核心数据模型定版
- 共享协议字段调整
- 初始化链路重构
- 全局状态机重构
- 跨模块集成收口
- 波次总验收
- 发布候选 go/no-go 判断

适合并行执行的工作包括：

- 边界清晰的独立模块实现
- 测试补齐
- 文档与审计证据整理
- 独立 UI / 交互优化
- 独立性能专项
- 独立安全专项
- 独立发布流水线增强

如果当前运行环境支持多子 agent，并且满足 `docs/step/94`、`96`、`98`、`99` 的前提条件，则优先采用并行车道执行。每个并行车道必须明确：

- 负责范围
- 主写入目录
- 输入依赖
- 输出交付包
- 阻塞条件
- 合并条件
- 验证标准

如果并行条件不成立，则沿用相同车道模型串行推进，避免伪并行导致反复返工。

### 八、每轮必须完成的落地动作

每轮执行时，必须直接做真实落地动作，不允许只输出“建议后续怎么做”。

至少必须完成以下事项中的适用项：

1. 修改代码、测试、脚本、配置、工作流、文档，使当前 Step 真正向 `L4` 推进。
2. 补齐与风险等级匹配的验证：
   - 单元测试
   - 集成测试
   - E2E / 冒烟测试
   - 性能验证
   - 安全回归验证
   - 安装 / 构建 / 打包 / 发布 / 回滚验证
3. 在 `docs/review/` 产出本轮证据，至少覆盖：
   - 本轮目标
   - 对应 Step / 波次
   - 实际改动
   - 风险与问题
   - 测试与验证结果
   - 当前 `L` 级别
   - 未完成项
   - 下一轮输入
4. 如本轮改动影响架构事实、能力边界、模块职责、数据流、流程、接口、性能结论、安全基线、测试门禁、发布方式或实施进度，必须按 `docs/step/97` 回写 `docs/架构/`。
5. 更新 `docs/release/` 下的版本信息与 changelog，确保本轮变更可追溯。

### 九、`docs/release/` 版本与 changelog 管理规则

如果 `/docs/release/` 不存在，先创建它。至少维护以下文件：

- `docs/release/CHANGELOG.md`
- `docs/release/最新版本说明.md`

在需要记录正式版本、里程碑版本、候选发布版本时，可新增：

- `docs/release/YYYY-MM-DD-vX.Y.Z-发布说明.md`

版本规则：

- 使用语义化版本：`MAJOR.MINOR.PATCH`
- 在尚未形成正式候选版本前，可在 `CHANGELOG.md` 顶部使用 `Unreleased` 暂存区；一旦形成候选发布，必须立即落为具体语义化版本
- 架构级不兼容调整、发布基线重定义、恢复策略重定义，可提升 `MAJOR`
- 新增一组可交付能力、完成一个重要 Step / 波次、形成候选版本，可提升 `MINOR`
- 缺陷修复、优化、补测、文档校正、发布流程修补，可提升 `PATCH`
- 所有日期必须使用绝对日期格式：`YYYY-MM-DD`
- 每次存在真实代码 / 测试 / 文档 / 脚本 / 构建 / 发布变更时，必须更新 changelog，不允许遗漏

changelog 书写标准：

- 使用专业、客观、可审计语言
- 必须按以下分类记录适用变更：
  - `Added`
  - `Changed`
  - `Fixed`
  - `Refactored`
  - `Performance`
  - `Security`
  - `Tests`
  - `Docs`
  - `Build`
  - `CI`
  - `Release`
- 每条记录都要说明：
  - 影响范围
  - 改动目的
  - 风险控制
  - 验证结果摘要
  - 关联 Step / 波次 / 架构能力
- 禁止写成“优化若干”“修复一些问题”“调整细节”这种不可审计表述

### 十、自我评估与自我迭代规则

每轮收尾前，必须进行自评，并据此决定是否继续迭代当前 Step。自评维度至少包括：

- 需求与产品对齐度
- 架构一致性
- 模块边界清晰度
- 组件复用与可维护性
- 数据一致性与恢复能力
- 性能
- 安全
- 测试完整性
- 发布就绪度
- 商业化可交付度

每个维度都必须给出以下四档之一：

- `未达标`
- `基本达标`
- `达标`
- `领先`

评估规则：

- 只要存在关键维度为 `未达标`，就不能宣称当前 Step 完成。
- 只要存在关键维度长期停留在 `基本达标`，就必须继续打磨，不能把“勉强可用”当作商业化完成。
- 自评必须有证据，不能主观自夸。
- 评估结果必须反映到 `docs/review/` 和必要的 `docs/架构/`、`docs/release/` 中。

### 十一、标准自我思考执行算法

每轮必须按以下算法自我思考并执行，不允许跳过：

1. 重新获取真实现状：
   - 重新扫描代码、测试、脚本、文档、Git 状态、最近变更证据。
2. 识别最大差距：
   - 判断当前最影响 `L4` 闭环、波次验收或商业化交付的最大缺口是什么。
3. 分类差距：
   - 该差距属于架构缺口、实现缺口、测试缺口、验证缺口、发布缺口、性能缺口、安全缺口还是文档缺口。
4. 判断最优策略：
   - 是继续补齐当前方案；
   - 还是先做边界修正；
   - 还是切换到更小闭环；
   - 还是转入并行车道；
   - 还是必须先止损修复回退。
5. 定义本轮最小闭环增量：
   - 明确本轮做完后，哪些能力、证据、评分、文档、发布记录会得到实质提升。
6. 直接实施：
   - 修改代码、测试、脚本、文档、工作流或配置，不允许停留在推理层。
7. 运行验证：
   - 运行与风险等级匹配的测试和验证，记录结果。
8. 比较前后状态：
   - 对比本轮前后的 `L` 级别、`91` 评分、关键风险数量、未完成项数量、回退数量、发布就绪度。
9. 作出下一步决策：
   - 继续打磨当前 Step；
   - 进入下一 Step；
   - 进入波次总验收；
   - 或进入持续优化模式。
10. 完成回写：
   - 更新 `docs/review/`、`docs/架构/`、`docs/release/`，确保结果可审计。

### 十二、连续迭代升级与策略切换规则

为了避免机械重复和无效迭代，必须执行以下升级规则：

- 如果同一个 Step 连续两轮以上仍未显著提升 `L` 级别或关键评分，必须输出根因分析，并切换实施策略。
- 如果连续两轮都在补同一类问题，必须判断其上游原因是否是边界设计错误、测试设计错误、验证口径错误或并行拆分错误。
- 如果当前 Step 已达到 `L4`，但仍存在显著的性能、安全、体验或发布短板，则不能直接宣称“完美完成”，必须转入持续优化清单。
- 如果当前 Step 的缺陷已经超出本 Step 设计边界，必须显式回写到：
  - `docs/架构/` 作为架构偏差或新增约束；
  - `docs/step/` 作为后续 Step 输入；
  - `docs/review/` 作为风险与遗留项；
  - `docs/release/` 作为版本风险说明。
- 如果当前策略已经开始制造大面积返工、共享冲突或验证阻塞，必须立即降级并行度，优先恢复稳定主脊柱。
- 如果当前版本已经具备候选发布条件，但仍存在若干“可优化但不阻塞上线”的问题，必须将其沉淀为明确 backlog，而不是无限延迟发布结论。

### 十三、商业化终局验收门

在追求“完美结果”时，最终必须经过以下商业化验收门。只要任一关键门未通过，就不能宣称达到终局状态：

1. 产品门：
   - 主路径完整、关键功能可用、体验一致、核心场景无致命缺口。
2. 架构门：
   - 模块边界清晰、职责稳定、依赖方向正确、无明显架构漂移。
3. 数据门：
   - 权威源、缓存、副本、同步、恢复、版本历史语义清晰。
4. 性能门：
   - 关键指标有量化结果、无明显主线程卡顿、可持续回归。
5. 安全门：
   - 会话、配置、桌面能力、发布链、敏感数据处理符合安全基线。
6. 质量门：
   - 单测、集成、E2E、性能、安全、发布验证覆盖适用范围。
7. 发布门：
   - 构建、安装、打包、签名、校验、发布、回滚链路可验证。
8. 运维门：
   - 关键日志、错误定位、故障隔离、回滚与恢复路径明确。
9. 文档门：
   - `docs/架构/`、`docs/step/`、`docs/review/`、`docs/release/` 与当前实现一致。
10. 商业化门：
   - 当前版本已具备对外或对内候选交付判断能力，风险、限制、遗留项和后续路线透明可控。

### 十四、最终完成标准

只有同时满足以下条件，才能宣称某个 Step、某个波次或某个版本候选真正完成：

- 对应能力达到 `L4`
- `docs/step/91` 审计总分达到可放行标准，且无一票否决项
- 代码、测试、验证、文档、集成全部闭环
- `docs/review/` 有完整证据
- `docs/架构/` 与真实实现保持一致
- `docs/release/` 已更新版本与 changelog
- 如属于波次完成，已执行 `docs/step/93` 波次总验收
- 如属于发布候选，已形成：
  - go/no-go 结论
  - 风险列表
  - 回滚策略
  - 遗留 backlog
  - 下一轮计划
- 如属于“接近完美”结论，必须说明：
  - 哪些维度已达 `领先`
  - 哪些维度仅为 `达标`
  - 哪些内容被刻意留在下一轮 backlog
  - 为什么这些遗留项不阻塞当前候选版本成立

### 十五、每次输出必须包含

每次执行完成后，输出中必须明确给出：

- 当前 Step
- 当前波次
- 当前 `L` 级别
- 本轮识别出的最大差距
- 本轮采用的实施策略与原因
- 本轮为何优先处理该项
- 本轮实际完成的代码 / 测试 / 文档 / 脚本 / 配置变更
- 本轮新增或更新的 `docs/review/`、`docs/架构/`、`docs/release/` 文件
```
