# 快捷键运行时补充设计

- 日期: 2026-04-07
- 所属 Step: 04
- 主题: workspace hotkey runtime coordinator

## 1. 背景

在此前实现中，`NotesWorkspacePage.tsx` 直接承担了以下热键运行时职责：

1. 绑定/解绑 `window` 的 `keydown`
2. 读取搜索焦点状态
3. 调用 `resolveNotesWorkspaceHotkeyCommand()`
4. 根据结果决定是否 `preventDefault()`
5. 分发到页面命令执行入口

这会使页面层同时承担显示、命令装配与事件运行时编排三类职责。

## 2. 当前设计

### 2.1 resolver 层

文件: `packages/sdkwork-canvas-canvas/src/services/noteWorkspacePageActions.ts`

职责:

- 定义热键输入到页面命令的纯函数映射

### 2.2 runtime 层

文件: `packages/sdkwork-canvas-canvas/src/services/noteWorkspaceHotkeyRuntime.ts`

职责:

- 绑定/解绑 `keydown`
- 调用 resolver
- 统一处理 `preventDefault()`
- 把命中的命令分发给页面命令执行入口

### 2.3 页面层

页面层只保留依赖装配：

1. 注入 `isCommandPaletteOpen`
2. 注入 `isSearchFocused()`
3. 注入 `resolveNotesWorkspaceHotkeyCommand`
4. 注入 `handleWorkspacePageCommand`

## 3. 流程说明

```text
window keydown
  -> noteWorkspaceHotkeyRuntime
      -> read search focus
      -> resolveNotesWorkspaceHotkeyCommand(...)
      -> if command exists
          -> preventDefault()
          -> execute page command
```

## 4. 架构价值

### 4.1 高内聚

热键运行时编排集中在单一 service，页面不再散落 keydown 监听细节。

### 4.2 低耦合

热键规则与热键绑定分层明确。后续要改规则，只动 resolver；要改运行时，只动 runtime。

### 4.3 易扩展

后续扩展以下能力时，可以继续沿当前边界推进：

- 平台特定热键兼容策略
- 编辑器模式热键分层
- 埋点与审计
- 多窗口或多作用域热键开关

## 5. 评估标准

| 维度 | 标准 | 当前状态 |
| --- | --- | --- |
| 运行时收口 | keydown 绑定必须脱离页面内联 effect | 达成 |
| 规则分层 | 解析规则与事件绑定必须分层 | 达成 |
| 可测试性 | hotkey runtime 必须可用 Node contract 独立验证 | 达成 |
| 页面复杂度 | 页面不得再内联 `preventDefault + dispatch` 分支 | 达成 |
| 剩余工作 | sidebar resize runtime 仍待下沉 | 未完成 |

## 6. 对标行业先进标准

成熟笔记应用的热键体系通常要求：

1. 快捷键规则稳定、可审计。
2. 运行时绑定可替换、可扩展。
3. 页面组件不承担底层事件编排责任。

当前实现已经在本项目内完成“规则纯函数化 + 运行时 service 化”的组合边界，为后续编辑器级热键体系与多平台适配打下基础。
