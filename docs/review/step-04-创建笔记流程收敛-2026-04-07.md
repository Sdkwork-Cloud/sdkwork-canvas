# Step 04 创建笔记流程收敛评审

- 日期: 2026-04-07
- Step: 04
- 当前能力等级: L3
- 评审范围: create note 页面级异步流程、默认标题解析、当前文件夹透传、成功后的视图回切

## 1. 评审结论

本轮增量将 `NotesWorkspacePage.tsx` 中内联的创建笔记流程下沉到独立运行时服务 `noteWorkspaceCreateNoteRuntime.ts`。页面层不再直接负责：

1. 根据笔记类型解析默认标题。
2. 组合当前选中文件夹上下文。
3. 判断创建结果是否成功。
4. 在创建成功后切回 `all` 视图。

本次收敛使“创建笔记”从零散页面动作提升为可测试、可复用、可扩展的页面级 runtime coordinator。

## 2. 设计与实现评审

### 2.1 设计合理性

- 优点: 创建笔记流程拥有了独立 service 边界，页面不再内联维护异步分支。
- 优点: 默认标题解析与父文件夹透传被统一到同一运行时入口，避免按钮入口、命令入口和后续快捷键入口各自拼装参数。
- 优点: `runTransition + setActiveView('all')` 的成功态切换规则被稳定封装，后续扩展到 telemetry、最近创建定位、草稿初始化时不需要再回到页面层重写。

### 2.2 实现逻辑评审

`noteWorkspaceCreateNoteRuntime.ts` 统一提供：

1. `createNotesWorkspaceCreateNoteRuntime(dependencies)`
2. `runtime.createNote(noteType)`

行为约束如下：

- 必须先按 `noteType` 解析默认标题。
- 必须把当前 `selectedFolderId` 原样透传给 store `createNote()`。
- 只有 `createNote()` 返回有效 note id 时，才允许执行视图切换。
- 创建失败或未返回 id 时，禁止触发 transition 和 `setActiveView('all')`。

### 2.3 架构质量评估

- 高内聚: 创建笔记流程集中在单一 runtime service，达到。
- 低耦合: 页面仅负责注入依赖与触发调用，达到。
- 易测试: 默认标题、父目录、成功态切换均可通过 Node contract 独立验证，达到。
- 易扩展: 后续为创建后自动聚焦、埋点、模板注入预留了稳定入口，基本达到。

## 5. 当前缺口

Step 04 仍维持 `L3`，原因如下：

1. `createNote` 的页面级流程已收敛，但 store 内部的 `createNote / createFolder / moveNote / renameFolder` 写路径协同仍未全部下沉为独立 service 边界。
2. 页面层仍保留 `workspace hotkey` 与 `sidebar resize` 两类运行时编排。
3. repository 仍未进入 read-through / replica / sync 策略抽象阶段。

## 6. 下一步建议

建议下一轮优先级：

1. hotkey runtime coordinator
2. sidebar resize runtime coordinator
3. 持续压缩 store 内的剩余写路径即时协同逻辑
