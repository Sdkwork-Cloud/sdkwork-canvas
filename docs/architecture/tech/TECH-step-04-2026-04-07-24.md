# Step 04 页面命令运行时收敛 - 2026-04-07

## 1. 本轮目标

在不改变当前业务行为的前提下，继续收紧 `Step 04-工作区边界收敛与数据访问抽象` 的页面容器边界，将 `NotesWorkspacePage.tsx` 中仍然内联的页面命令依赖组装、命令执行以及命令面板/对话框命令转换，收敛为可复用、可验证、可审计的独立 runtime service。

本轮目标不是新增功能，而是把页面命令编排中枢从页面组件内部抽离，进一步降低页面容器对业务编排的直接耦合。

## 2. 实施范围

本轮实际落地文件如下：

- `sdkwork-canvas-pc-react/packages/sdkwork-canvas-canvas/src/services/noteWorkspacePageCommandRuntime.ts`
- `sdkwork-canvas-pc-react/packages/sdkwork-canvas-canvas/src/services/index.ts`
- `sdkwork-canvas-pc-react/packages/sdkwork-canvas-canvas/src/pages/NotesWorkspacePage.tsx`
- `sdkwork-canvas-pc-react/scripts/workspace-page-command-runtime.contract.test.mjs`
- `sdkwork-canvas-pc-react/package.json`
- `sdkwork-canvas-pc-react/scripts/package-scripts-contract.test.mjs`

## 3. 设计与实现结论

### 3.1 新增页面命令运行时边界

新增 `noteWorkspacePageCommandRuntime.ts`，统一承载以下能力：

- 接收页面命令依赖工厂输入
- 生成稳定的页面命令执行运行时
- 执行显式页面命令
- 将命令面板 action 解析为页面命令后执行
- 将对话框确认事件解析为页面命令后执行

该运行时本身不负责 UI 渲染、不负责 Zustand 状态管理、不负责浏览器事件绑定，而是作为页面容器与既有命令依赖工厂/命令执行器之间的装配边界。

### 3.2 页面侧收口结果

`NotesWorkspacePage.tsx` 本轮完成以下收口：

- 不再内联 `createNotesWorkspacePageCommandDependencies()` 与 `executeNotesWorkspacePageCommand()` 的组合调用
- 不再在页面内直接处理命令面板 action 到页面命令的转换
- 不再在页面内直接处理对话框确认事件到页面命令的转换

页面当前仅保留：

- 浏览器 ref 与焦点适配
- 组件装配
- 少量 icon / layout / status 展示胶水

这使页面容器更接近“视图装配层”，而不是“业务编排层”。

## 4. TDD 与验证证据

node --test --experimental-test-isolation=none scripts/package-scripts-contract.test.mjs

### 4.3 合同覆盖点

`workspace-page-command-runtime.contract.test.mjs` 已覆盖：

- 运行时统一组装页面命令依赖并执行显式页面命令
- `change-view / open-note / toggle-inspector` 等命令的真实执行链路
- 命令面板 action 到页面命令的转换与执行
- 对话框确认事件到页面命令的转换与执行

## 5. 架构收益评估

### 5.1 直接收益

- 页面命令编排不再散落在 `NotesWorkspacePage.tsx` 中
- 既有 `page action -> dependency factory -> executor` 链路完成运行时级收口
- 命令面板与对话框确认路径共享同一执行边界，避免页面重复拼装

### 5.2 对后续 Step 的支撑

- 为继续压缩页面容器展示胶水提供稳定中间层
- 为 Step 04 剩余页面边界清理建立“runtime service + Node contract”复用模式
- 为后续桌面命令桥接、批量动作与更复杂命令来源提供统一扩展位

## 6. 评估结论

本轮增量完成后，Step 04 的“页面命令运行时编排”已从页面内部逻辑中抽离为独立 service，页面职责继续收敛，测试与脚本门禁同步到位，但 **Step 04 仍维持 `L3`**，原因如下：

- store 写路径即时状态协调仍未完全下沉
- repository 策略接缝尚未建立
- 页面容器仍保留少量展示与装配胶水

## 7. 下一轮建议

建议严格按以下顺序继续：

1. 优先清理 `renameFolder / moveNote / createNote / createFolder` 的写路径协调残留。
2. 建立 repository 的 read-through / replica / queued-sync 策略接缝。
3. 最后继续压缩页面容器中的 icon / layout / status 展示胶水。
