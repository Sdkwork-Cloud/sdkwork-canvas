# Step 04 - 文件夹选择回退收敛

- 日期：`2026-04-07`
- Step：`04-工作区边界收敛与数据访问抽象`
- 本轮主题：`currentSelectedFolderId refresh fallback`
- 当前结论：`初始化链中的文件夹选择回退已进入 orchestrator，但 Step 04 仍未达到 L4`

## 1. 本轮目标

在上一轮完成 `trash` 视图列表源回退后，本轮继续处理工作区初始化链中的第二个关键残留：

- `selectedFolderId` 不应继续由 store 在刷新后自行猜测。
- orchestrator 必须显式判断：
  - 当前文件夹是否仍然存在；
  - 当前视图是否允许保留文件夹选择；
  - 缺失或不适用时是否应回退为 `null`。

## 2. 实际改动

### 2.1 orchestrator 初始化结果扩展

`packages/sdkwork-canvas-canvas/src/services/noteWorkspaceOrchestrator.ts`

- `initializeWorkspace()` 新增 `currentSelectedFolderId` 输入。
- `NoteWorkspaceInitializationState` 新增 `selectedFolderId` 输出。
- orchestrator 现在会根据以下规则决策文件夹选中态：
  - 当前视图为 `trash` -> `selectedFolderId = null`
  - 当前文件夹仍存在 -> 保留原值
  - 当前文件夹已不存在 -> 回退为 `null`

### 2.2 store 接线收口

`packages/sdkwork-canvas-canvas/src/store/useNotesWorkspaceStore.ts`

- `initialize()` 调用 orchestrator 时显式传入 `currentSelectedFolderId`。
- store 应用 orchestrator 返回的 `selectedFolderId`，不再把刷新后的文件夹有效性判断完全留在本地状态层。

### 2.3 Node 契约测试

`sdkwork-canvas-pc-react/scripts/workspace-orchestrator.contract.test.mjs`

- 新增文件夹选择回退契约测试，覆盖：
  - 文件夹仍存在时保留选中；
  - 文件夹缺失时回退为 `null`；
  - `trash` 视图时强制清空文件夹选择。

## 3. 结果判断

本轮之后，工作区初始化链已经具备以下可验证事实：

- `currentActiveView`
- `currentSelectedFolderId`
- `activeNoteId`

以上三类输入都已开始通过 orchestrator 统一编排，而不再完全依赖 store 的默认状态推断。

## 4. 仍未完成的缺口

Step 04 仍不能升到 `L4`，因为还有以下残留：

1. 删除文件夹、移动文件夹等变更后的即时状态清理仍有一部分停留在 store。
2. 页面层仍保留快捷键与 action dispatch 编排细节。
3. repository 仍缺少 `read-through / replica / sync` 的策略级抽象接口。
