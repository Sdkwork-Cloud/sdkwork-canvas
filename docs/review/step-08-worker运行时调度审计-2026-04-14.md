## 本轮已冻结的运行时语义

1. runtime 会串行 drain 队列，直到没有 runnable task。
2. 重叠的 `requestDrain()` 请求会合并到同一个 active run，不会并发执行多个 worker。
3. runtime 只会为最早到期的 `retrying` 任务挂起 timer，并在到期后自动回放。
4. timer 到期后的回放仍然复用同一套 executor 状态机，不会引入第二套转移语义。
5. `dispose()` 会取消未到期 timer，避免释放后的幽灵回放。

## 风险与剩余缺口

- 当前 runtime 仍是 package-local API，不是 `canvas-canvas` 或 desktop 的真实后台调度线程。
- 当前 `execute(task)` 仍是注入式抽象，没有真实远端 transport、鉴权与幂等保护。
- 当前没有 UI 层冲突提示或手动 replay 入口，用户恢复闭环仍未形成。
- 当前没有离在线切换 smoke 与冲突演练记录，无法把 `CP08-4` 整体提升到 `L3/L4`。

## 下一轮建议

1. 优先把 runtime 接到真实 handler 边界，而不是先写 UI。
2. 在 runtime 接好后定义 remote ack 应用语义，明确成功回执如何更新 cursor 与本地状态。
3. 以最小成本补冲突恢复入口或离在线 smoke，把 `CP08-4` 从“已启动”推向“已形成集成闭环”。
