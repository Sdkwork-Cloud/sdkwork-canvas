# Step 03 - 会话安全与配置治理升级

## 1. 目标与范围

本 step 用于收敛当前应用最突出的基础风险：会话令牌存储、配置源解析、owner scope 推断、远端设置 hydration 规则和平台差异下的安全回退。

### 1.1 执行输入

- Step 02 骨架结果
- `docs/架构/04`、`05`、`06`、`08`
- `canvas-core/useAppSdkClient.ts`
- `canvas-shell/AppProviders.tsx`
- `canvas-desktop` 桌面桥和 capability

### 1.2 本步非目标

- 不在本 step 内实现本地数据库
- 不在本 step 内实现同步队列或全文检索

### 1.3 最小输出

- 会话安全迁移方案与适配层
- 配置治理边界与优先级规则
- 远端设置 hydration 的稳定行为

## 2. 架构对齐

- `docs/架构/04-技术选型与可扩展策略.md`
- `docs/架构/05-数据模型与存储设计.md`
- `docs/架构/06-业务流程-应用接口与集成设计.md`
- `docs/架构/08-安全-测试-安装-部署-发布设计.md`

## 3. 当前现状与问题

当前已知问题：

- 高敏令牌仍依赖 `localStorage`
- SDK client 配置源较灵活，但还未形成正式治理边界
- 远端设置 hydration 已有实现，但失败恢复与去重逻辑需要稳定化和文档化

## 4. 设计

### 4.1 会话安全目标

- 刷新令牌进入安全存储或明确受控的兼容层
- 访问令牌尽量短期化和内存化
- Web 和 Desktop 保持统一接口、不同底层适配

### 4.2 配置治理目标

- 收敛 `process.env + import.meta.env + __SDKWORK_CANVAS_ENV__` 的解析优先级
- 固化 `ownerMode = root / tenant / organization` 的推断和作用域选择
- 避免页面直接消费环境变量

## 5. 实施落地规划

1. 在 `canvas-core` 抽象安全会话存储接口
2. 在 `canvas-desktop` 提供桌面安全存储适配实现
3. 收敛 SDK 配置解析、ownerMode 与平台默认值
4. 收敛远端 settings hydration 去重、失败回退与测试

## 6. 测试计划

- `useAppSdkClient` 单测
- `AppProviders` hydration 单测
- capability / desktop bridge 测试
- 环境配置 contract test

## 7. 结果验证

完成后必须满足：

- 会话安全不再只是散落在 `localStorage`
- 平台、ownerMode 和环境配置的行为可预测、可测试
- hydration 失败不会阻断应用壳

### 8.6 本 Step 完成后必须兑现的架构能力

- 敏感会话存储从高风险持久层迁移到更安全的治理边界
- 配置分层、来源优先级、敏感项处理和环境差异全部冻结
- Web/Desktop 在会话刷新、退出清理和故障恢复上保持一致策略

### 8.7 最快完成的并行执行顺序

2. 再并行推进安全存储适配与迁移兼容逻辑
3. 最后统一补登录/刷新/登出/恢复测试和文档回写
4. 会话模型未冻结前，不允许工作区链路直接接入新的安全实现

## 9. 风险与回滚

### 9.1 风险

- 迁移令牌存储时若不保留兼容层，会影响现有登录态恢复

### 9.2 回滚

- 保留受控兼容读取期，逐步迁移后再移除旧 key
