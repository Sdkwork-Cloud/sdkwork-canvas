# Step 04 自动保存运行时收敛评审

- 日期: 2026-04-07
- Step: 04
- 当前能力等级: L3
- 评审范围: autosave timer 调度、`pagehide` 绑定、页面副作用装配收敛

## 1. 评审结论

本轮增量在前一轮 autosave strategy 抽离的基础上，继续将自动保存运行时装配从 `NotesWorkspacePage.tsx` 下沉到独立服务 `noteWorkspaceAutosaveRuntime.ts`。页面层不再直接手写 `setTimeout` 与 `pagehide` 监听的装配细节，而是通过 runtime service 统一调度。

这次收敛使自动保存链路形成更完整的双层边界：

1. strategy 层负责判断何时应该自动保存。
2. runtime 层负责如何装配 timer 与 pagehide 绑定。

## 2. 设计与实现评审

### 2.1 设计合理性

- 优点: strategy 与 runtime 分层更清晰，不再把“何时保存”和“如何装配副作用”混在页面内。
- 优点: runtime service 仍保持依赖注入，不直接依赖 React Hook 或 DOM 全局对象。
- 优点: 可以为未来的编辑器自动保存、离线草稿、visibilitychange flush 等扩展保留统一入口。

### 2.2 实现逻辑评审

`noteWorkspaceAutosaveRuntime.ts` 统一提供了：

1. `scheduleNotesWorkspaceAutosave(plan, deps)`
2. `bindNotesWorkspacePageHideAutosave(plan, deps)`

行为约束如下：

- 仅当 plan 开启调度且 `delayMs` 存在时才注册 timer。
- 仅当 plan 要求 pagehide flush 时才注册页面隐藏监听。
- 不满足条件时统一返回 noop cleanup，避免页面分支散落。

### 2.3 架构质量评估

- 高内聚: autosave runtime 行为集中在单一服务，达成。
- 低耦合: 页面不再感知 timer/pagehide 绑定细节，达成。
- 易测试: timer 与 pagehide 均可通过依赖注入独立 contract 化，达成。
- 易扩展: visibilitychange、beforeunload、queue flush 等可沿同一路径扩展，基本达成。

## 4. 评估标准与达成情况

| 评估项 | 标准 | 结果 |
| --- | --- | --- |
| 运行时边界 | 页面不得直接编排 autosave timer 与 pagehide 监听 | 达成 |

## 5. 当前缺口

Step 04 仍维持 `L3`，原因如下：

1. create note flow 仍在页面直接协调。
2. hotkey 监听与 sidebar resize runtime 仍在页面层。
3. 页面虽然已显著收敛，但尚未完全达到 runtime coordinator 全部外置的程度。
