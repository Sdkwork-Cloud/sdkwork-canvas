# 04. 技术选型与可扩展策略

## 4.1 当前技术栈概览

### 4.1.1 客户端与桌面技术栈

| 技术 | 当前用途 | 评价 |
|---|---|---|
| React 19 | UI 和交互层 | 先进且合适 |
| Vite 8 | 前端开发与构建 | 先进且高效 |
| TypeScript | 类型安全 | 必选基础设施 |
| Tailwind CSS 4 | 视觉系统与样式组织 | 适合快速构建设计系统 |
| Zustand | 轻量本地状态管理 | 适合当前结构 |
| TanStack Query | 远端状态和缓存 | 方向正确，但尚未充分发挥 |
| Tiptap / ProseMirror 生态 | 富文本编辑能力 | 专业且可扩展 |
| Sonner | 全局 toast 与交互反馈 | 轻量且适合桌面工作台反馈 |
| Tauri 2 | 桌面宿主层 | 对当前产品形态非常匹配 |
| Rust | 桌面运行时、托盘、配置与未来本地能力载体 | 具有长期潜力 |
| package workspace + Turbo | 多包工程与并行构建 | 工程化选择正确 |

### 4.1.2 远端集成技术

| 技术 | 当前用途 | 评价 |
|---|---|---|
| `@sdkwork/app-sdk` | 对接 note、filesystem、user、auth | 正确，应继续坚持 SDK 优先 |
| `@sdkwork/auth-pc-react` | 统一认证体验与控制器 | 显著降低认证重复建设成本 |
| `sdkwork.app.config.json` + `app:update-create` | 应用元数据、发布目标与产物模板治理 | 体现了比普通前端项目更强的交付治理意识 |

### 4.1.3 平台与制品拓扑

| 维度 | 当前事实 | 价值 |
|---|---|---|
| Web 入口构建 | 根应用 `src/main.tsx` 通过主 `vite.config.ts` 产出 Web 入口 | 保留浏览器访问与 Web bundle 交付能力 |
| Desktop 入口构建 | `packages/sdkwork-canvas-desktop` 通过独立 Vite/Tauri 配置构建桌面壳 | 将平台壳与共享业务壳解耦 |
| 平台变量注入 | 根 `vite.config.ts` 默认 `VITE_APP_PLATFORM=web`，桌面包默认 `VITE_APP_PLATFORM=desktop`，也支持显式覆盖 | 让 SDK 与运行时逻辑按平台感知行为 |
| 发布元数据 | `sdkwork.app.config.json` 声明 `web-production`、`desktop-windows`、`desktop-macos`、`desktop-linux` 等包 | 明确区分访问 URL、下载包和平台类型 |
| Runtime payload | `artifacts.runtimePayloads.TAURI.manifestUrl` 按环境声明更新清单地址 | 为未来自动更新闭环预留标准入口 |

## 4.2 当前选型是否合理

结论是：**合理，而且基础足够先进。**

原因如下：

- React + Vite 非常适合桌面工作台型应用，开发效率和生态成熟度高。
- Tiptap 能支撑专业编辑器的长期扩展，而不是停留在简单 textarea。
- Tauri + Rust 比传统 Electron 更利于控制桌面能力边界、性能和后续本地能力扩展。
- package workspace + Turbo 让拆包和持续演进更容易治理。
- 通过共享 SDK 接入远端能力，避免前端业务包直接耦合协议细节。
- Web 与 Desktop 共享 `AppRoot`，使平台差异主要停留在启动壳和桥接层，而不是业务实现层。

当前并不是“技术栈落后”，而是“技术栈先进，但能力架构尚未用满”。

---

## 4.3 当前技术短板不在框架，而在能力层缺口

即使技术栈先进，仍有以下缺口：

| 缺口 | 本质问题 |
|---|---|
| 本地优先 | 当前没有本地数据库和同步队列 |
| 搜索 | 当前没有本地全文索引引擎 |
| 协作 | 当前没有 CRDT/操作日志/协作协议 |
| 安全 | 当前已完成会话分层与配置治理，但尚未进入 OS 级密钥存储与更强的前端安全基线 |
| 观测 | 当前没有日志、指标、错误收敛体系 |
| 平台化 | 当前没有插件、自动更新、发布渠道治理 |

因此后续升级重点应是“能力层补齐”，而不是“重做前端技术栈”。

### 4.3.1 Step 02 骨架预埋现状

截至 `2026-04-07`，当前工作区已经把以下目标能力从“架构规划”推进为“真实工程骨架”：

- `@sdkwork/canvas-local`
- `@sdkwork/canvas-search`
- `@sdkwork/canvas-sync`
- `@sdkwork/canvas-observability`
- `@sdkwork/canvas-updater`

这些包当前只提供最小导出面和未来责任边界，不代表能力已经实现。其工程价值在于：

- 先冻结能力落点，再推进后续实现
- 避免本地优先、搜索、同步、观测和更新继续堆入 `canvas-canvas`
- 让 package workspace + Turbo 能从一开始把未来包图纳入内部工程治理

### 4.3.2 Step 03 当前技术决议

截至 `2026-04-07`，Step 03 已把“会话安全与配置治理”从目标态推进为当前真实技术决议：

| 主题 | 当前决议 | 说明 |
|---|---|---|
| Web 会话存储 | `sessionStorage` | 浏览器端默认把聚合会话快照保存在 `sessionStorage`，不再以 `localStorage` 作为主持久层 |
| 兼容迁移 | 一次性 `localStorage -> sessionStorage` 迁移 | 启动时读取历史 `sdkwork-canvas-auth-session` 与旧 token 键，迁移后立即清理旧键 |
| Desktop 会话存储 | `canvas-desktop` 原生会话桥 + `desktop-session.json` | 前端侧镜像保持在 `sessionStorage`，桌面权威副本通过 Tauri 命令读写 app data 目录 |
| 配置来源优先级 | `__SDKWORK_CANVAS_ENV__ > import.meta.env > process.env` | `canvas-core` 已导出稳定的运行时上下文解析结果，避免页面层自行拼环境变量 |
| Owner Mode 治理 | `root / tenant / organization` 冻结为显式运行时语义 | 优先读取 `VITE_APP_OWNER_MODE` 等显式配置，不存在时再根据租户/组织信号推断 |
| 平台识别 | `override > Tauri runtime > env default` | Desktop 运行时默认识别为 `desktop`，Web 默认识别为 `web`，同时保留显式覆盖能力 |

这意味着当前最紧急的会话与配置风险已经从“高风险散落存储”降为“受控但仍可继续加固”的状态。下一阶段不应重新发明另一套会话/配置层，而应在现有边界上继续推进工作区初始化、数据访问和本地优先能力。

## 4.4 面向行业领先的目标选型建议

### 4.4.1 本地存储与索引

| 能力 | 建议选型 | 原因 |
|---|---|---|
| 本地数据库 | SQLite / libSQL（由 Rust 层统一封装） | 桌面端成熟、可靠、易迁移 |
| 全文检索 | SQLite FTS5 或 Tantivy | 适合本地全文检索和结构化检索 |
| 附件缓存 | 本地文件系统 + 校验元数据 | 支持离线资产与恢复 |
| 队列与日志 | Rust 持久化任务队列 | 适合后台同步和恢复 |

### 4.4.2 同步与协作

| 能力 | 建议选型 | 原因 |
|---|---|---|
| 增量同步 | 游标 + 变更日志 + 幂等重放 | 易于和现有远端平台集成 |
| 协作编辑 | Yjs 或 Automerge + 服务端中继 | 适合富文本和离线合并 |
| 版本历史 | 文档快照 + 操作日志混合 | 同时兼顾恢复与成本 |

### 4.4.3 安全与平台能力

| 能力 | 建议选型 | 原因 |
|---|---|---|
| 密钥存储 | OS Keychain / Windows Credential Manager / macOS Keychain | 不应长期使用 `localStorage` 保存高敏感会话 |
| 自动更新 | Tauri updater + 渠道签名治理 | 桌面产品必备 |
| Feature Flags | 本地 + 远端配置联合治理 | 支持灰度和风险隔离 |
| 崩溃监控 | 统一 error reporting 平台 | 支撑线上质量治理 |

## 4.5 可扩展策略

### 4.5.1 业务扩展策略

未来所有新增能力，应优先沿用以下扩展点：

- **编辑器扩展点**：围绕 Tiptap extension 增加块类型、引用、数据库块、卡片块。
- **命令扩展点**：命令面板应成为统一动作入口，而不是每个页面自建快捷入口。
- **领域服务扩展点**：新增同步、搜索、AI、协作都应作为独立服务域存在。
- **桌面能力扩展点**：所有系统能力仍通过 `canvas-desktop` 桥接暴露。
- **主题与语言扩展点**：统一走 design tokens 和 i18n 资源，不接受局部硬编码。

### 4.5.2 包级扩展策略

新增包时应遵循：

- 如果能力跨多个业务域复用，放到 shared 层。
- 如果能力有独立领域模型，建独立领域包。
- 如果能力只服务平台壳，放到 `canvas-desktop` 或未来平台包。
- 不允许为了赶进度把搜索、同步、AI 直接塞进 `canvas-canvas` 页面组件。
- 当某个目标能力已经拥有独立包骨架后，后续实现默认必须在该包内继续推进，除非架构文档明确改口。

### 4.5.3 协议与 SDK 扩展策略

- 继续坚持 SDK 优先，不引入手写 HTTP 孤岛。
- 远端能力新增时，先补 SDK 与契约，再做前端消费。
- 本地能力新增时，优先以服务接口抽象，避免页面直接依赖实现细节。

## 4.6 技术选型评估标准

每一项新增技术都应回答以下问题：

| 评估项 | 核心问题 |
|---|---|
| 适配性 | 是否真的适合桌面优先的笔记产品，而不是流行即合理 |
| 生命周期 | 未来 3-5 年是否可维护、可升级 |
| 集成成本 | 是否能接入现有 workspace、Tauri、Rust、SDK 体系 |
| 性能收益 | 是否能显著改善启动、搜索、同步、稳定性 |
| 安全影响 | 是否引入新的攻击面、权限面、供应链风险 |
| 退出成本 | 如果选错，是否可迁移、可替换 |
