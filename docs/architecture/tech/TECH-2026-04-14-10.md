# 最新版本说明 - 第五十轮推进补记 - 2026-04-14

本轮继续停留在 `Step 08 / CP08-4`，目标不是新增远端同步能力，而是把工作区 store 从“模块加载即固定实例”推进为“可在 bootstrap 期装配的运行时边界”。这一轮完成后，`canvas-canvas` 工作区已经具备在 app/bootstrap 或 desktop/background 层注入自定义 store/runtime 的基础能力，但仍然没有越界去伪造远端 handler 或 ack apply。

## 本轮完成

1. `useNotesWorkspaceStore.ts` 增加 live store binding 与 bootstrap API：
   - `getNotesWorkspaceStore()`
   - `setNotesWorkspaceStore(store)`
   - `configureNotesWorkspaceStore(overrides?)`
   - `resetNotesWorkspaceStore()`
2. `useNotesWorkspaceStore(...)` 改为 wrapper hook，始终读取当前导出的 store。
3. 新增 `workspace-store-bootstrap.contract.test.mjs`，锁定 store bootstrap 的替换、读取、reset 行为。
4. 对齐 `workspace-sync-runtime-boundary`、`workspace-sync-write-path`、`workspace-startup-recovery-smoke` 的 zustand stub，保证新导出形态在 Node contract 环境可验证。

## 当前状态

- `Step 08 = L2`
- `CP08-4 / 冲突与失败恢复验证 = L2`
- `CP08-4 / canvas-canvas 工作区 store bootstrap 装配边界 = L3`

## 还缺什么

- 真实远端执行 handler。
- ack apply 与 `remoteCursor` 闭环。
- app shell 或 desktop/background 对 store/runtime 的正式 bootstrap 接线。

## 结论

这轮解决的是“后续真实同步运行时从哪里装进去”，不是“同步闭环已经完成”。因此 Step 不升级，仍然继续在 `Step 08 / CP08-4` 内推进。
