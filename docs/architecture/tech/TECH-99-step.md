# 99. Step 极速并行执行总作战手册

## 1. 文档定位

本文件不是替代 `94 / 96 / 98`，而是把三者收敛成一份可以直接执行的“总作战手册”。

目标只有两个：

- 在风险可控前提下，以最快速度完成 `00-13`
- 确保每个 Step 执行完毕后，确实兑现其对应的架构能力，而不是只完成局部代码改动

## 2. 核心原则

极速推进并不意味着无序并行，而是坚持以下公式：

`主脊柱串行冻结 + 波次内主写入范围并行 + 独立验证车道 + 波次级总验收`

如果缺少其中任意一项，速度越快，返工越大。

## 3. 先决条件

在按本手册执行前，以下文档必须已经可用：

- `docs/step/README.md`
- `docs/step/90-架构能力-Step-代码目录-证据映射矩阵.md`
- `docs/step/94-Step并行执行编排与车道拆分建议.md`
- `docs/step/95-架构能力闭环验收标准.md`
- `docs/step/96-Step并行执行周计划与排班建议.md`
- `docs/step/98-Step并行车道交付包与集成交接标准.md`

## 4. Step 完成即能力完整实现的判定公式

任何一个 Step，只有同时满足以下五项，才允许判定为“该 Step 已完整实现对应架构能力”：

1. 该 Step 文档 `8.6` 中列出的能力切片已兑现
2. 该 Step 文档 `8.7` 中列出的最速执行顺序已完成收口
3. `95` 的闭环标准达到 `L4`
4. `97` 的架构回写和能力兑现清单已输出
5. 该 Step 对下一 Step 的准入条件已冻结

只满足 1-2 项，不允许视为完成。

## 5. 当前仓库默认推荐模式

对当前 `sdkwork-canvas` 仓库，推荐默认使用：

`Fast-M`

原因：

- 相比 `Fast-S`，推进速度更快
- 相比 `Fast-L`，共享边界冲突更少
- 能同时覆盖主业务链、平台链和质量链
- 对当前 Notes 应用的包结构最匹配

## 6. 三档极速执行模式

### 6.1 `Fast-S`

适用：

- `2` 人或 `2` 个子 Agent
- 以稳定性优先，适合关键基线冻结阶段

编组：

- `Lane-S1`：主脊柱 / 集成 / 文档 / 验收
- `Lane-S2`：功能实现

适合波次：

- `Wave-A`
- `Wave-B` 前半段

### 6.2 `Fast-M`

适用：

- `4` 人或 `4` 个子 Agent
- 当前仓库的默认推荐模式

编组：

- `Lane-M1`：集成 / 审计 / 文档 / 最终收口
- `Lane-M2`：壳层 / 安全 / 配置治理
- `Lane-M3`：工作区初始化 / 编辑器 / 本地存储
- `Lane-M4`：搜索 / 同步 / 测试门禁

适合波次：

- `Wave-A`
- `Wave-B`
- `Wave-C`

### 6.3 `Fast-L`

适用：

- `6` 人或 `6` 个子 Agent
- 有成熟集成 Owner，且周中可多次集成

编组：

- `Lane-L1`：集成 / 架构 / 文档 / 发布结论
- `Lane-L2`：壳层 / 安全 / 配置治理
- `Lane-L3`：初始化 / 编辑器 / 本地存储
- `Lane-L4`：搜索 / 同步
- `Lane-L5`：桌面壳 / 发布链
- `Lane-L6`：测试 / 性能 / 版本恢复

适合波次：

- `Wave-C`
- `Wave-D`

## 9. 每轮并行执行固定算法

推荐每个波次、每个大 Step 都按下列算法运行：

1. 冻结本轮目标、边界、Owner、共享文件
2. 冻结统一验证命令与检查点
3. 拆车道并分配主写入目录
4. 周中或中期做一次集成检查
5. 统一执行验证车道
6. 输出审计、复盘、回写
7. 通过后再进入下一轮

这是本项目的默认执行算法，不建议跳步。

## 10. 每个 Step 最快完成的收口动作

每个 Step 的最快正确收口方式，固定如下：

1. 完成主目标实现
2. 对照 Step `8.6` 检查能力是否真正兑现
3. 对照 Step `8.7` 检查执行顺序是否完整闭环
4. 运行适用测试与验证
5. 输出 `docs/review/` 审计记录
6. 回写 `docs/架构/`
7. 冻结下一 Step 输入

任何缺少第 2、5、6、7 步的收口，都视为不完整。

## 11. 不允许并行的情形

以下情况不允许继续并行推进：

- 主写入目录和 Owner 未冻结
- 共享边界文件无单一拍板人
- 统一验证命令不存在
- 当前波次仍有 `B3/B4` 级阻塞
- 上一波次总验收未通过

## 12. 最容易失败的四类反模式

- 只并行实现，不并行验证
- 先写代码，后补边界
- 一个车道同时主写多个高风险共享文件
- Step 自称完成，但没有兑现其 `8.6` 能力切片

## 13. 推荐的实际落地节奏

若按默认推荐模式 `Fast-M` 执行，建议采用：

- 周一：冻结输入、Owner、边界、检查点
- 周二到周三：并行实现
- 周三晚：中期集成
- 周四：补测试、补验证、修冲突
- 周五：审计、回写、波次或 Step 收口

### 14.1 Step 放行三层门

任何 Step 要想正式放行到下一步，都必须同时通过以下三层门：

1. 功能门
   - 主实现完成
   - 检查点通过
2. 能力门
   - `8.6` 能力切片兑现
   - `95` 闭环等级达到 `L4`
3. 文档门
   - `97` 要求的回写、审计、兑现清单齐全

只过功能门，不过能力门或文档门，统一视为假完成。

## 16. 最快并行执行的硬规则

为了保证“快而不乱”，实际执行时必须再额外遵守以下规则：

- 一个车道只允许拥有一组主写入目录
- 一个共享边界文件只能有一个最终拍板 Owner
- `10` 不能滞后于 `04-09` 的主要实现
- 每一轮中期集成只处理冲突、偏移和验证滞后，不临时扩 scope
- 每个 Step 收口前必须同时经过 `91 + 95 + 97`
- 每个波次收口前必须经过 `93`

## 17. 结论

对当前 Notes 应用，最快且完整的执行方式不是“把 00-13 全部分散并行”，而是：

- 用 `00-03` 冻结主脊柱
- 用 `Fast-M` 作为默认并行模式
- 在 `Wave-C` 和 `Wave-D` 需要提速时，升级到 `Fast-L`
- 用 `95 + 97 + 99` 三份规则确保每个 Step 收口时真正兑现架构能力

如果严格按本手册推进，才能在“快”和“完整”之间取得真实平衡，而不是用短期速度换长期返工。
