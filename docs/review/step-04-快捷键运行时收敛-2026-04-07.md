# Step 04 快捷键运行时收敛评审

- 日期: 2026-04-07
- Step: 04
- 当前能力等级: L3
- 评审范围: workspace keydown 绑定、热键命令解析入口、preventDefault 与页面命令分发

## 1. 评审结论

本轮增量将 `NotesWorkspacePage.tsx` 中内联的 `keydown` 监听、快捷键命令解析调用与命令分发编排下沉到独立运行时服务 `noteWorkspaceHotkeyRuntime.ts`。页面层不再直接负责：

1. 绑定/解绑 `window` 的 `keydown` 监听。
2. 将 `KeyboardEvent` 归一化为热键解析输入。
3. 根据命令是否命中决定是否 `preventDefault()`。
4. 将解析结果转交给 `handleWorkspacePageCommand()`。

这使工作区热键从页面临时逻辑升级为可独立验证的 runtime 边界。

## 2. 设计与实现评审

### 2.1 设计合理性

- 优点: 热键解析规则仍留在 `noteWorkspacePageActions.ts`，而运行时绑定与事件分发进入 `noteWorkspaceHotkeyRuntime.ts`，职责拆分清晰。
- 优点: runtime 只消费依赖注入，不直接依赖 React Hook 或浏览器全局单例，测试成本低。
- 优点: 搜索焦点状态判断和命令面板开启状态都通过依赖传入，避免把页面状态耦合写死在 service 中。

### 2.2 实现逻辑评审

`noteWorkspaceHotkeyRuntime.ts` 统一提供：

1. `bindNotesWorkspaceHotkeys(dependencies)`

行为约束如下：

- runtime 必须统一绑定 `keydown` 监听，并返回 cleanup。
- 每次事件都必须重新读取当前搜索焦点状态。
- 只有 resolver 返回有效命令时，才允许执行 `preventDefault()` 与命令分发。
- 未命中命令时，必须透明放行浏览器默认行为。

### 2.3 架构质量评估

- 高内聚: 热键运行时编排集中到单一 service，达到。
- 低耦合: 页面仅负责注入 resolver、状态读取与命令执行函数，达到。
- 易测试: keydown 绑定、焦点状态透传、preventDefault 和命令执行均可 contract 化，达到。
- 易扩展: 后续增加编辑器特定快捷键、平台特定修饰键策略、埋点统计时具备稳定入口，基本达到。
