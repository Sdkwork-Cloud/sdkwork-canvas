# 06-业务流程-应用接口与集成设计-串行保存编排补充

- 日期：`2026-04-13`
- 关联 Step：`Step 05`
- 关联波次：`Wave-B / 第二十九轮推进`
- 关联能力：`保存编排层 / save queue`

## 1. 背景

在完成“统一刷盘入口”和“失败反馈与重试状态机”后，当前产品仍暴露出保存链的第四个关键结构问题：所有保存触发器虽然都已共享 `flushDraft`，但真实持久化执行仍缺少串行编排层，导致连续保存、高风险切换和旧响应回写三个场景都可能继续损伤草稿一致性。

该问题会直接影响后续三项能力：

1. 自动退避与最大重试次数设计
2. 页面关闭 / 异常退出前的最终刷盘可靠性
3. Step 06 本地草稿与崩溃恢复的接入边界

## 2. 当前真实实现

### 2.1 保存编排服务

`noteWorkspaceSaveQueue.ts` 当前提供统一的 `NotesWorkspaceSaveQueue`：

| API | 含义 | 当前语义 |
| --- | --- | --- |
| `hasActiveRequest()` | 是否存在 in-flight save | 用于区分“新请求需要 replay”还是“只需 wait” |
| `waitForActiveRequest()` | 等待当前请求完成 | 高风险切换在 `saving / retrying` 时必须调用 |
| `requestReplay()` | 请求当前保存完成后再跑一次 | 连续编辑期间的二次保存只合并为一次 replay |
| `run(save)` | 以串行队列执行真实保存 | 同一时间只允许一个活跃请求 |

当前设计约束：

1. replay 只做合并，不无限排队。
2. 当前请求失败后，不继续自动 replay，避免隐藏重试风暴。
3. queue 只处理编排，不直接决定 UI 反馈与文案。

### 2.2 保存完成语义

`resolveNotesWorkspaceSaveCompletion()` 当前统一处理保存成功后的状态落地：

1. 总是先基于请求快照与返回 summary 生成 `persistedActiveNote`
2. 若当前 `activeNote` 已出现更新编辑：
   - 只更新 `persistedActiveNote`
   - 保持 `activeNote` 为更新后的本地草稿
   - 将 `saveState` 重新置回 `dirty`
3. 若当前 `activeNote` 仍与请求快照一致：
   - 才把 `activeNote` 更新为保存成功后的版本

这条规则明确冻结了一个重要现实：保存成功不再等价于“当前编辑器草稿可以直接回退到请求快照”。

### 2.3 Store 集成边界

`useNotesWorkspaceStore.ts` 当前已形成如下边界：

1. `persistUnsavedActiveNoteIfNeeded()`
   - `dirty / error`：立即 flush
   - `saving / retrying`：等待 active request
2. `persistActiveNote()`
   - active request 存在且当前仍为 `dirty / error`：调用 `requestReplay()`
   - active request 存在且当前不是 `dirty / error`：调用 `waitForActiveRequest()`
   - 没有 active request：调用 `activeNoteSaveQueue.run(save)`
3. 高风险动作
   - `loadActiveNote`
   - `createNote`
   - `deleteFolder`
   - `moveNoteToTrash`
   - `moveNote`
   以上动作都必须先经过 `persistUnsavedActiveNoteIfNeeded()`

### 2.4 当前集成关系

1. `autosavePlan`
   - 决定何时允许 schedule / flush
2. `saveFeedback`
   - 决定用户看到什么状态
3. `saveQueue`
   - 决定真实保存请求如何串行、等待和 replay
4. `store`
   - 负责调用 `workspaceService.save()` 并更新工作区状态

## 3. 本轮设计收益

### 3.1 并发保存可解释

现在可以明确解释三种常见场景：

1. 用户连续输入触发多次 autosave
   - 同时只会有一个远端请求
   - 最多追加一次 replay
2. 用户在保存中继续编辑
   - 当前请求完成后会回放一次新保存
   - 更新草稿不会被旧响应覆盖
3. 用户在保存中触发高风险动作
   - 不再误判为“非 dirty 因而直接放行”
   - 必须先等待当前请求完成

### 3.2 后续扩展成本下降

自动退避、重试上限、观测埋点、Step 06 本地草稿恢复，都可以围绕统一 `saveQueue` 扩展，而不需要再次侵入页面和编辑器层。

### 3.3 集成假设更稳定

Step 06 若要引入本地草稿或崩溃恢复，只需要接管“保存失败后的本地保底路径”，不需要重写当前串行保存和高风险等待语义。

## 4. 评估标准

### 4.1 设计评估标准

1. 保存链是否有独立的串行编排层，而不是把队列状态散落在 store 分支里。
2. replay 是否明确建模为“最多一次后续保存”，而不是无限队列或隐式重入。
3. 旧响应回写规则是否已显式冻结。

### 4.2 实现评估标准

1. `persistActiveNote()` 必须消费 `saveQueue`。
2. 高风险动作在 `saving / retrying` 时必须等待，而不是放行。
3. 保存成功后不能继续直接用请求快照覆盖当前更新草稿。

### 4.3 测试评估标准

1. `workspace-save-queue.contract.test.mjs` 必须冻结 queue API、replay 合并与 stale response protection。
2. `workspace-high-risk-flush-boundary.contract.test.mjs` 必须冻结 `dirty / error -> flush`、`saving / retrying -> wait`。
3. 既有 `workspace-save-feedback.contract.test.mjs` 与 `workspace-save-flush-boundary.contract.test.mjs` 必须继续成立。

### 4.4 集成评估标准

1. 新 contract 必须进入 `test:workspace:contracts`。
2. `package-scripts-contract.test.mjs` 必须同步冻结新的脚本聚合链。
3. 根级 `pnpm.cmd typecheck` 必须继续跑通，不允许 save chain 演进破坏其他包。

## 5. 当前结论

“串行保存编排”这一增量能力已经达到 `L4`，但 Step 05 整体当前只推进到 `L3`：

1. `flushDraft` 单入口已冻结
2. `error -> retrying -> recovered` 状态机已冻结
3. `saveQueue` 串行编排与高风险等待已冻结
4. 但自动退避、重试上限、保存观测与异常退出证据仍未形成稳定集成约束

因此当前最准确的判断是：

- `Step 05 = L3`
- 仍不得进入 `Step 06`

## 6. 下一轮设计约束

1. 自动退避若引入，必须建立在 `saveQueue` 上，而不是新增第二套并发控制。
2. Step 06 的本地草稿恢复必须消费当前 `persistedActiveNote / activeNote / saveState` 语义，不得重新定义“旧响应是否可以覆盖当前草稿”。
3. 任何新的高风险动作，只要可能丢失或替换当前草稿，都必须先复用 `persistUnsavedActiveNoteIfNeeded()`，再进入真实业务逻辑。
