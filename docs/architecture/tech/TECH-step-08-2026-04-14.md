## 本轮已冻结的执行语义

1. worker 只消费 oldest runnable queued task，不并发拉取多条任务。
2. `retrying` 任务只有在 `nextRetryAt <= execution.at` 时才会被释放回 `queued`。
3. handler 调用前必须先持久化 `running`，避免执行中断时队列状态失真。
4. 成功回执可以刷新 `remoteCursor`，且不会丢失既有 cursor。
5. retryable failure 统一回写为 `retrying`，terminal failure 统一回写为 `failed`。
6. conflict 统一进入 `conflict` 状态，保留手动恢复入口的语义位置。

## 风险与剩余缺口

- 当前 executor 仍是包内 API，不是实际 background worker runtime。
- 当前 handler 仍是注入式抽象，没有真实远端 transport、鉴权与幂等保护实现。
- 当前没有 UI 层冲突提示或失败恢复入口，因此用户层恢复闭环仍未形成。
- 当前没有离在线切换 smoke 与冲突演练记录，无法把 `CP08-4` 整体提升到 `L3/L4`。

## 下一轮建议

1. 在不扩大 UI 面的前提下，把 executor 接到真实运行时调度边界。
2. 定义并验证远端回执应用语义，明确成功回执如何更新 cursor 与本地状态。
3. 补最小冲突恢复入口和离在线切换 smoke，推动 `CP08-4` 从 `L2` 进入 `L3`。
