# Step 05 串行保存编排审计

- 日期：`2026-04-13`
- 阶段：`Step 05 / L3`
- 波次：`Wave-B / 第二十九轮推进`
- 本轮主题：`save queue 串行编排与高风险刷盘等待`

## 1. 本轮目标

本轮承接上一轮已经完成的“失败反馈与重试状态机”能力，继续解决保存链中的下一个主阻塞：虽然 `flushDraft` 已成为唯一保存入口，但连续保存请求仍可能并发交叠，高风险动作在 `saving / retrying` 期间也缺少明确的等待语义，导致保存链还不具备真正稳定的串行编排能力。

本轮聚焦的真实缺口如下：

1. `persistActiveNote()` 在已有 in-flight save 时，无法显式区分“等待当前请求完成”和“要求当前请求完成后立即回放一次新保存”。
2. 切换笔记、创建笔记、删除文件夹、移入回收站等高风险动作，在 `saveState === 'saving' / 'retrying'` 时可能直接越过保存链，而不是等待当前请求落稳。
3. 旧请求返回时会把较早的请求快照重新写回 `activeNote`，存在覆盖更新本地草稿的风险。

## 2. 实际完成

1. `sdkwork-canvas-pc-react/packages/sdkwork-canvas-canvas/src/services/noteWorkspaceSaveQueue.ts`
   - 新增 `createNotesWorkspaceSaveQueue()`，冻结 `hasActiveRequest / waitForActiveRequest / requestReplay / run` 四类串行编排能力。
   - 新增 `resolveNotesWorkspaceSaveCompletion()`，把“旧保存响应返回时如何保护更新草稿”的收口逻辑从 store 中抽离为独立 helper。
   - 当前队列语义为：同一活跃笔记仅允许一个远端保存请求在飞行；若飞行期间再次触发保存，只合并为一次 replay；若当前请求失败，不自动继续 replay。
2. `sdkwork-canvas-pc-react/packages/sdkwork-canvas-canvas/src/store/useNotesWorkspaceStore.ts`
   - 新增 `activeNoteSaveQueue = createNotesWorkspaceSaveQueue()`，把活跃笔记保存链收敛到单一串行队列。
   - `persistUnsavedActiveNoteIfNeeded()` 改为：
     - `saveState === 'dirty' / 'error'` 时立即 flush
     - `saveState === 'saving' / 'retrying'` 时等待当前请求完成
   - `persistActiveNote()` 改为：
     - 若已有 active request 且当前仍是 `dirty / error`，则调用 `requestReplay()`
     - 若已有 active request 且当前不是 `dirty / error`，则调用 `waitForActiveRequest()`
   - 保存成功后改为统一消费 `resolveNotesWorkspaceSaveCompletion()`：
     - 仅在当前草稿仍与请求快照一致时，把 `activeNote` 更新为保存后的版本
     - 若本地已有更新编辑，则只更新 `persistedActiveNote`，并保持 `activeNote` 为更新草稿且 `saveState = dirty`
3. `sdkwork-canvas-pc-react/scripts/workspace-save-queue.contract.test.mjs`
   - 新增 save queue contract。
   - 冻结“串行保存 + replay 合并 + 旧响应不覆盖新草稿 + store 必须消费 save queue”四项架构事实。
4. `sdkwork-canvas-pc-react/scripts/workspace-high-risk-flush-boundary.contract.test.mjs`
   - 升级高风险刷盘 contract。
   - 冻结“`dirty / error` 必须 flush、`saving / retrying` 必须 wait”这一新边界。
5. `sdkwork-canvas-pc-react/package.json`
   - 将 `workspace-save-queue.contract.test.mjs` 与 `workspace-high-risk-flush-boundary.contract.test.mjs` 纳入 `test:workspace:contracts` 聚合链。
6. `sdkwork-canvas-pc-react/scripts/package-scripts-contract.test.mjs`
   - 同步冻结新的 contract 聚合链，避免脚本门禁回退。

## 3. Contract 驱动与验证结果

   - 仓库中不存在独立的 save queue 服务

   - 旧逻辑只覆盖 `dirty / error`，没有显式要求 `saving / retrying` 等待当前请求完成

## 4. 架构判断

本轮完成后，Step 05 的保存链职责边界进一步稳定为：

1. `noteWorkspaceAutosave.ts` 与 `noteWorkspaceAutosaveRuntime.ts`
   - 负责“何时可以 schedule / flush”以及 `visibilitychange(hidden) / pagehide` 等触发器装配。
2. `noteWorkspaceSaveFeedback.ts`
   - 负责“用户当前看到什么保存状态”的可见反馈模型。
3. `noteWorkspaceSaveQueue.ts`
   - 负责“真实保存请求如何串行执行、如何合并 replay、如何等待 in-flight request、如何保护更新草稿”。
4. `useNotesWorkspaceStore.ts`
   - 负责真实持久化调用、状态落地与高风险动作前的一致性控制。

这意味着 Step 05 现在已经同时具备：

1. 单一 `flushDraft` 入口
2. 显式 `error -> retrying -> recovered` 反馈状态机
3. `save queue` 串行编排与 replay 合并
4. 高风险动作前的 `dirty/error -> flush`、`saving/retrying -> wait` 规则
5. 旧响应不覆盖更新草稿的保护逻辑
