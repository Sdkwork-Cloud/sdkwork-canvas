# Step 04 对话框运行时收敛评审

- 日期: 2026-04-07
- Step: 04
- 当前能力等级: L3
- 评审范围: destructive dialog 运行时协调、确认执行链路、回收站恢复流程

## 1. 评审结论

本轮增量在已有 dialog state 抽离的基础上，继续将对话框相关运行时逻辑从 `NotesWorkspacePage.tsx` 下沉到独立服务 `noteWorkspaceDialogRuntime.ts`。页面层不再直接维护“打开清空回收站/删除笔记/删除文件夹对话框”“确认后关闭并执行命令”“恢复回收站笔记后切回 all 视图并选中”这些流程细节。

这次收敛的价值比上一轮更高，因为它处理的不再只是展示态，而是明确收拢了用户动作到副作用执行的运行时闭环。

## 2. 设计与实现评审

### 2.1 设计合理性

- 优点: `pendingDialog` 相关行为由独立 runtime service 协调，页面只保留状态持有与渲染。
- 优点: destructive action 的打开、关闭、确认执行形成单一入口，避免后续新增对话框类型时继续向页面追加分支。
- 优点: 恢复笔记流程和对话框确认流程都采用依赖注入，便于 contract 测试与后续跨端复用。

### 2.2 实现逻辑评审

`createNotesWorkspaceDialogRuntime()` 统一收拢了以下行为：

1. `openClearTrashDialog()`
2. `openDeleteNoteDialog(noteId)`
3. `openDeleteFolderDialog(folderId)`
4. `closeDialog()`
5. `confirmDialog(pendingDialog)`
6. `restoreNote(noteId)`

其中：

- `confirmDialog()` 会先关闭 dialog，再执行命令，符合当前页面行为。
- `restoreNote()` 在恢复成功后切回 `all` 视图并选中恢复后的笔记，失败时不触发额外副作用。

### 2.3 架构质量评估

- 高内聚: destructive dialog runtime 集中在单一服务，达成。
- 低耦合: 页面不再直接编排恢复与确认逻辑，达成。
- 易测试: 纯依赖注入 + Node contract，达成。
- 易扩展: 新增 destructive dialog 类型时，可继续在 runtime service 中扩展，基本达成。

## 4. 评估标准与达成情况

| 评估项 | 标准 | 结果 |
| --- | --- | --- |
| 运行时边界 | 页面不得直接维护 destructive dialog 打开/关闭/确认执行细节 | 达成 |
| 恢复流程 | restore 成功后必须稳定切回 all 并选中目标笔记 | 达成 |
| 失败行为 | restore 失败不得触发多余 transition/select | 达成 |
| 契约接入 | 新 runtime service 必须纳入 workspace contract 聚合 | 达成 |
| 页面降复杂 | 页面内与对话框运行时相关的处理函数必须减少 | 达成 |

## 5. 当前缺口

Step 04 依然应维持 `L3`，理由如下：

1. autosave runtime 的 `setTimeout/pagehide/flush` 生命周期编排仍在页面内。
2. create note 流程仍由页面直接协调。
3. hotkey 监听与 sidebar resize runtime 仍保留在页面层。

## 6. 下一步建议

建议下一轮优先级如下：

1. autosave runtime coordinator
2. create note flow coordinator
3. sidebar resize runtime 抽离

其中以 autosave runtime coordinator 优先级最高，因为它同时符合 Step 05 的编辑器可靠性方向，也能继续压缩页面副作用装配复杂度。
