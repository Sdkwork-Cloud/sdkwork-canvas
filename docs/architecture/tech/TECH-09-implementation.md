# 09. 实施计划

## 9.1 计划目标

本计划的目标不是笼统地“做更多功能”，而是把 SDKWork Canvas 从“优秀的桌面笔记工作台”推进为“具备行业领先潜力的桌面优先知识应用”。

计划必须满足四个约束：

- 每个阶段都可交付、可验证、可回退。
- 每个阶段都明确涉及模块、交付物和退出门禁。
- 先补基础硬标准，再扩能力半径。
- 任何阶段完成后，`README + 01-10` 都要同步更新，保持文档与代码一致。

---

## 9.2 实施前提

在进入具体阶段前，先冻结以下前提：

- 当前主工程为 `sdkwork-canvas-pc-react`，不得并行另起一套新前端架构。
- 当前主数据来源仍是远端 `@sdkwork/app-sdk`，所有本地优先能力都要以“增量引入本地能力层”的方式演进。
- 当前最重要的风险不是 UI 不够漂亮，而是会话安全、本地能力缺失、观测不足和高层自动化测试不足。
- 当前最重要的机会不是横向堆功能，而是把安全、性能、搜索、同步和恢复打成平台底座。

### 9.2.1 当前 Wave-B 执行注记（`2026-04-13`）

- `Step 04` 已完成 `L4`，`repository / orchestrator / selector / page` 主链边界已经冻结。
- `Step 05` 当前推进到 `L3`，已经形成 `flushDraft + save feedback + save queue` 的保存主脊柱。
- 阶段二后续工作可以默认依赖以下事实：
  - 连续保存请求已串行化
  - 高风险动作在 `saving / retrying` 时必须等待当前请求完成
  - 旧保存响应不会覆盖更新后的本地草稿
- 在进入 `Step 06` 前，仍需补齐自动退避、重试上限、保存观测以及页面关闭/异常退出证据矩阵。

## 9.3 阶段划分

### 阶段一：基础硬化

**目标**

- 提升会话安全、质量门禁和桌面启动稳定性。
- 为后续本地能力层、同步层和搜索层打地基。

**涉及模块**

- `canvas-auth`
- `canvas-core`
- `canvas-shell`
- `canvas-desktop`
- `scripts/*`
- `.github/workflows/*`

**核心任务**

| 任务 | 说明 | 交付物 |
|---|---|---|
| 会话安全升级 | 刷新令牌迁移到 OS 安全存储，清理当前高风险 `localStorage` 方案 | 安全存储适配层、迁移方案、回退方案 |
| 启动链观测 | 建立启动、认证就绪、工作区就绪、保存链打点 | 指标点位、错误归因、耗时统计面板 |
| E2E 骨架 | 覆盖登录、工作区、编辑、回收站、账户页主路径 | E2E 测试基础设施与 smoke 套件 |
| 视觉回归基线 | 锁定认证页、工作区、账户页、深浅主题快照 | 视觉基线仓库与更新流程 |
| 发布可信链一期 | 建立签名、SBOM、产物校验与来源证明路线 | 发布安全设计文档和流水线门禁 |
| 配置治理梳理 | 明确 `.env`、远端设置、本地偏好的边界 | 配置清单、敏感项治理标准 |

**退出门禁**

- `verify-release` 与本地开发主门禁持续全绿。
- 主路径 E2E 可以稳定通过。
- 新版本 crash-free sessions 已可观测。
- 敏感会话信息不再以当前形式长期暴露在 `localStorage`。

### 阶段二：本地优先与搜索

**目标**

- 让产品具备离线可用、大规模可搜索和可恢复的本地能力。

**涉及模块**

- `canvas-canvas`
- `canvas-core`
- `canvas-search`（新增）
- `canvas-sync`（新增）
- `canvas-desktop`

**核心任务**

| 任务 | 说明 | 交付物 |
|---|---|---|
| 本地数据库引入 | 建立笔记、文件夹、偏好、本地草稿结构化存储 | 本地存储 schema 与迁移机制 |
| 全文索引 | 为正文、标题、标签、视图统一建立本地索引 | 搜索引擎、索引构建器、查询接口 |
| 工作区初始化重构 | 从“远端扫描 + 客户端聚合”升级为“首屏优先 + 增量加载” | 新初始化编排与性能基准数据 |
| 同步队列一期 | 引入待同步队列、同步状态和失败重试 | 最小同步状态机 |
| 崩溃恢复 | 未同步草稿和异常退出恢复 | 恢复策略与恢复入口 |

**退出门禁**

- 断网场景仍可打开和编辑最近工作区。
- 10k 笔记搜索 `P95 < 150ms`。
- 工作区初始化明显摆脱“全量扫描感”。
- 命令面板与搜索入口开始共享同一索引来源。

### 阶段三：版本、协作与知识发现

**目标**

- 从单人工作台升级到可恢复、可共享、可发现的知识系统。

**涉及模块**

- `canvas-canvas`
- `canvas-sync`
- `canvas-search`
- `canvas-collab`（新增）
- `canvas-ai`（新增）

**核心任务**

| 任务 | 说明 | 交付物 |
|---|---|---|
| 版本历史 | 支持历史版本查看、差异比较和恢复 | 版本模型、版本 UI、恢复流程 |
| 共享与权限模型 | 建立空间、共享链接、角色权限和审计基础 | 权限模型、分享策略、审计点位 |
| 协作底座 | 支持评论、presence、冲突处理或 CRDT 路线 | 协作协议与最小产品闭环 |
| 双向链接与引用 | 支持关系发现和反向链接 | 关系索引、引用 UI、导航链路 |
| AI 辅助一期 | 支持摘要、结构整理、问答检索等能力 | AI 服务边界与用户入口 |

**退出门禁**

- 用户能查看和恢复历史版本。
- 共享和权限可用且可审计。
- 知识发现不再只依赖文件夹和标签。
- 新增协作能力不会破坏离线和同步基本盘。

### 阶段四：平台化与行业领先运营能力

**目标**

- 建立可长期演进、可治理、可扩展的平台化能力。

**涉及模块**

- `canvas-desktop`
- `canvas-observability`（新增）
- `canvas-plugin-runtime`（新增）
- 发布与配置治理脚本

**核心任务**

| 任务 | 说明 | 交付物 |
|---|---|---|
| 自动更新体系 | 建立多渠道发布、灰度和回滚能力 | 更新策略、渠道配置、回滚脚本 |
| 观测平台 | 打通错误、日志、指标、业务分析 | 统一观测模型和仪表盘 |
| 插件机制 | 提供受控扩展点和沙箱隔离 | 插件 API、生命周期和权限模型 |
| 企业能力 | 规划 SSO、审计、策略、私有化预案 | 企业路线图与能力边界 |
| AI/知识网络深化 | 深化智能搜索、建议和自动化工作流 | 进阶智能能力路线 |

**退出门禁**

- 发布后具备自动验收和回滚能力。
- 运行状态可被量化观测。
- 新能力可通过插件或 Feature Flag 受控上线。

## 9.5 组织与工程治理建议

为确保计划可执行，建议明确以下责任面：

| 角色 | 责任 |
|---|---|
| 产品负责人 | 需求分层、版本目标、用户价值排序 |
| 客户端负责人 | Shell、Notes、User、Desktop 架构和体验质量 |
| 平台负责人 | SDK、同步、搜索、本地数据库、发布链 |
| 质量负责人 | 测试体系、质量门禁、回归策略 |
| 安全负责人 | 会话、权限、发布可信链和安全审查 |

同时建立以下治理动作：

- 每阶段开始前冻结入口条件和退出门禁。
- 每阶段结束前更新 `10-实施进度` 与相关主题文档。
- 所有新增模块必须定义 Owner、公开面、测试边界和回退方案。

## 9.6 通用验收标准

每个阶段都要同时满足五类验收：

| 类别 | 验收要求 |
|---|---|
| 功能验收 | 用户路径完整可演示，且现状与目标边界清晰 |
| 质量验收 | 测试、类型、构建、桌面验证全绿 |
| 性能验收 | 关键指标达标，有数据支撑 |
| 安全验收 | 不引入新的高风险存储、权限和发布漏洞 |
| 文档验收 | `README + 01-10` 与源码事实同步更新 |

## 9.7 实施计划结论

SDKWork Canvas 不需要“推翻重做”。最优路径是：

1. 先硬化安全、测试、观测和启动质量。
2. 再建设本地优先、搜索和同步地基。
3. 再进入版本、协作、知识发现和 AI 阶段。
4. 最后完成自动更新、插件、企业能力与平台化治理。

这条路线能保证每一步都建立在当前优秀工程基础上，而不是重新发明一套架构。

## 9.8 执行治理与基线审计回写（2026-04-07）

基于 `docs/step/00`、`docs/step/01`、`docs/step/02` 与 `docs/step/03` 的实际执行，本计划补充冻结以下现实结论：

- `docs/review/` 已建立为 Step 审计、能力兑现、质量审计与决议记录的统一证据目录。
- `Step 00` 已完成执行门禁、证据目录、回滚口径和 review 命名规范冻结。
- `Step 01` 已完成当前 packages、scripts、workflow 与脏工作树现状的基线审计。
- `Step 02` 已完成未来能力包骨架预埋，当前真实 workspace 已扩展为 `14` 个内部包，并在 `canvas-shell` 中冻结共享壳层与未来能力包边界清单。
- `Step 03` 已完成会话安全、配置治理与桌面运行时会话桥接收口：
  - `canvas-core` 已冻结统一会话存储适配接口与运行时配置解析入口。
  - Web 会话默认进入 `sessionStorage`，旧 `localStorage` 键只做一次性迁移后清理。
  - Desktop 会话由 Tauri 命令桥接到 app data 目录中的 `desktop-session.json`，前端保留 `sessionStorage` 镜像。
  - `.env.example` 与 `.env.development / .env.test / .env.production` 已显式声明 `VITE_APP_OWNER_MODE=tenant`。
- 当前 `scripts/` 下共有 `38` 个文件，说明 workspace / desktop / release 合同门禁已经继续增强。
- `pnpm.cmd test:workspace:contracts`、`pnpm.cmd test:desktop:contracts`、`pnpm.cmd test:desktop:rust`、`pnpm.cmd typecheck` 当前都可直接执行，`spawn EPERM` 已不再是有效阻塞项。
- 当前工作树仍处于活跃演进中的中间态，后续 Step 04 及之后的实施仍必须坚持渐进式收敛，避免覆盖既有业务改动。
- 当前 `packages/` 目录已经扩展为 `14` 个内部包，后续新能力默认必须优先在对应包内落地，而不是继续往 `canvas-canvas` 堆积。

基于以上结论，当前推荐的最优推进顺序更新为：

`04 -> 05 -> 06 -> 07 -> 08 -> 09 -> 10 -> 11 -> 12 -> 13`
