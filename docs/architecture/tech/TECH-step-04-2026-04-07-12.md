# Step 04 数据源能力契约收敛

- 日期：`2026-04-07`
- Step：`04-工作区边界收敛与数据访问抽象`
- 轮次：`Wave-B / 第四轮推进`
- 当前结论：`Step 04 保持 L3，新增“工作区数据源能力契约”主链路收敛`

## 1. 本轮目标

本轮聚焦补齐 Step 04 在“数据访问边界可描述性”上的缺口。此前工作区已经完成：

- `orchestrator` 初始化边界收敛
- `selector` 视图模型收敛
- `command palette` 模型收敛

但工作区快照仍缺少一份可显式表达当前实现现实的“数据源能力契约”，导致后续 Step 06/07/08 在本地副本、增量同步、离线能力、冲突处理等扩展方向上缺少统一扩展点。

## 2. 设计原则

本轮明确采用以下约束：

- 不伪造未来能力，契约只表达当前真实能力。
- 先固化事实，再为未来扩展预留字段，不提前引入本地副本实现。
- 契约必须贯通 `repository -> service -> orchestrator -> store`，不能只停留在单点类型定义。
- 契约必须进入 `test:workspace:contracts`，成为持续门禁的一部分。

## 3. 收敛结果

### 3.1 新增统一契约

在 `packages/sdkwork-canvas-canvas/src/types/canvasWorkspace.ts` 中新增：

- `NoteWorkspaceDataSource`
- `NoteWorkspaceDataSourceCapabilities`
- `createRemoteAppSdkNoteWorkspaceDataSource()`
- `createEmptyNoteWorkspaceSnapshot()`

当前工作区被精确定义为：

- `driver = app-sdk`
- `authority = remote`
- `readStrategy = workspace-snapshot`
- `writeStrategy = direct-write`

当前明确不支持：

- 本地副本
- 读穿缓存
- 离线读
- 离线写
- 后台同步
- 增量同步
- 冲突处理

### 3.2 契约主链路贯通

以下链路已完成同步：

- `noteRepository.queryWorkspaceSnapshot()` 显式返回 `dataSource`
- `noteWorkspaceService` 使用统一的 `NoteWorkspaceSnapshot` 类型
- `noteWorkspaceOrchestrator.initializeWorkspace()` 将 `dataSource` 纳入初始化结果
- `useNotesWorkspaceStore` 将 `dataSource` 纳入 store 状态

这意味着后续页面层、观察性埋点、同步模块、本地副本模块都可以读取同一份能力描述，而不需要继续从隐式实现猜测工作区当前模式。

### 3.3 契约测试入门禁

新增文件：

- `sdkwork-canvas-pc-react/scripts/workspace-data-source.contract.test.mjs`

覆盖两类断言：

- 默认数据源契约准确表达当前远端主导能力基线
- orchestrator 初始化结果原样透传 `dataSource`

同时更新：

- `sdkwork-canvas-pc-react/package.json`
- `sdkwork-canvas-pc-react/scripts/package-scripts-contract.test.mjs`

确保 `test:workspace:contracts` 将该契约作为标准门禁执行。

## 4. 价值判断

本轮不是简单“补字段”，而是完成了 Step 04 缺失的一块关键抽象：

- 让工作区边界具备“可描述、可验证、可演进”的能力
- 为 Step 06 本地副本、Step 07 搜索索引、Step 08 同步编排提供统一扩展入口
- 降低未来扩展时在 store/page 中散落硬编码判断的概率

## 6. 当前仍未关闭的 L4 缺口

本轮完成后，Step 04 仍保持 `L3`，原因如下：

- `refresh / list source / selection fallback` 仍有一部分责任留在 store
- 页面层仍保留部分快捷键编排与 action dispatch 细节
- repository 只完成了“能力契约显式化”，尚未进入 read-through / replica / sync 策略抽象

## 7. 下一轮建议

建议继续沿 Step 04 收口，不提前进入 Step 05：

1. 把 `refresh / list source / selection fallback` 进一步从 store 收敛到 orchestrator。
2. 把页面快捷键和 action dispatch 继续下沉为 page 之外的可测试边界。
3. 在不实现本地副本的前提下，补充 repository 的读写策略占位抽象，为 Step 06/08 做接口级预埋。
