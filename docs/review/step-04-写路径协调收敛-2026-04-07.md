# Step 04 写路径协调收敛 - 2026-04-07

## 1. 本轮目标

在不改变当前业务行为的前提下，继续收敛 `Step 04-工作区边界收敛与数据访问抽象` 的写路径边界，将以下即时状态协调从 `useNotesWorkspaceStore.ts` 中剥离为可复用、可验证、可审计的纯服务：

- `createNote`
- `createFolder`
- `renameFolder`
- `moveNote`

本轮目标不是新增功能，而是将“写入成功后的本地状态收敛逻辑”从 store 内联分支中抽离出来，形成统一的状态协调能力，为后续 page action dispatch 与 repository 读策略抽象继续降耦。

## 2. 实施范围

## 3. 设计与实现结论

### 3.1 新增纯服务能力

新增 `noteWorkspaceWriteCoordinator.ts`，统一承载以下纯函数：

- `planCreatedNoteState()`
- `planCreatedFolderState()`
- `planRenamedFolderState()`
- `planMovedNoteState()`

这些函数只接收当前状态快照与写入结果，输出下一状态，不依赖 store、不依赖 UI、不依赖副作用，从而满足：

- 高内聚：写路径即时状态收敛逻辑集中在同一服务中。
- 低耦合：store 只负责调用服务并提交状态，不再维护大量分支细节。
- 易验证：可通过 Node contract 直接验证，无需依赖浏览器或 Vite 运行时。
- 易扩展：后续增加 `duplicateNote`、`batchMove`、`restoreToFolder` 等能力时，可按同一模式扩展。

### 3.2 收敛的具体行为

#### 创建笔记

- 新建笔记后自动成为 `activeNote`
- 自动切回 `activeView = all`
- 自动同步 `selectedFolderId`
- 自动将目标父文件夹加入 `expandedFolderIds`
- 自动移除同 id 的废纸篓残留项
- 自动按 `updatedAt` 排序插入 `canvas`

#### 创建文件夹

- 新建文件夹后统一排序到 `folders`
- 若为子文件夹，自动展开父文件夹分支

#### 重命名文件夹

- 支持后端返回“重命名后 id 变化”的收敛场景
- 自动回写子文件夹 `parentId`
- 自动回写 `canvas / trashedNotes / activeNote` 的 `parentId`
- 自动回写 `selectedFolderId`
- 自动回写 `expandedFolderIds`

#### 移动笔记

- 自动更新 `canvas` 中目标笔记的 `parentId`
- 自动同步 `activeNote.parentId`
- 自动刷新 `updatedAt`
- 自动展开目标父文件夹

## 5. 架构收益评估

### 5.1 对当前产品的直接收益

- store 复杂度继续下降，写路径分支从“状态拼接”转为“服务编排”
- 工作区本地状态规则被显式化，降低未来重构回归风险
- 即时状态协调形成独立边界，后续可直接复用到命令面板动作、批量操作、桌面桥接动作

### 5.2 对后续 Step 的支撑

- 为 Step 04 剩余的 page action dispatch 下沉提供统一调用面
- 为 Step 06/08 的同步、离线、读策略扩展保留状态协调插槽
- 为更高等级的 contract-first 改造建立稳定模式

## 6. 评估结论

本轮增量完成后，Step 04 的“写路径即时状态协调”已从 store 内联逻辑中抽离出第一层稳定边界，质量状态继续提升，但 **Step 04 仍维持 `L3`**，原因如下：

- 页面层仍保留较多 hotkey / command / dialog action dispatch 编排逻辑
- repository 仍未形成 `read-through / replica / sync` 可扩展的读策略抽象
- store 虽已减薄，但仍承担部分页面动作编排职责

## 7. 下一轮建议

建议严格按以下顺序继续：

1. 提取 `NotesWorkspacePage.tsx` 的 action dispatch / hotkey orchestration 服务，并先补 Node contract。
2. 为 repository 增加读策略接口层，先不实现本地副本，只先建立可替换边界。
3. 完成上述两项后，再重新审计 Step 04 是否满足 `L4` 关闭条件。
