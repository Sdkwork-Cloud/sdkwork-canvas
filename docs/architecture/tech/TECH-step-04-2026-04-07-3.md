# Step 04 - 列表源与选中回退收敛

- 日期：`2026-04-07`
- Step：`04-工作区边界收敛与数据访问抽象`
- 本轮主题：`trash view refresh / list source / selection fallback`
- 当前结论：`完成一次真实边界下沉，但 Step 04 仍为 L3`

## 1. 本轮目标

本轮不继续扩展新功能，而是修正 Step 04 中一个影响初始化链精度的残留问题：

- 当工作区当前位于 `trash` 视图时，刷新后的选中回退不应再从正常笔记列表推导。
- 选中源必须和当前列表源一致，否则 orchestrator 仍不是真正的工作区编排边界。

## 2. 实际改动

### 2.1 orchestrator 契约扩展

`packages/sdkwork-canvas-canvas/src/services/noteWorkspaceOrchestrator.ts`

- `initializeWorkspace()` 新增 `currentActiveView` 输入。
- orchestrator 现在按当前视图决定选中回退的列表源：
  - `all` -> `canvas`
  - `favorites` -> 收藏列表
  - `recent` -> 最近列表
  - `trash` -> `trashedNotes`
- 当当前视图为 `trash` 且详情接口未返回实体时，仍沿用回收站摘要回退为只读详情。

### 2.2 store 接线收口

`packages/sdkwork-canvas-canvas/src/store/useNotesWorkspaceStore.ts`

- `initialize()` 调用 orchestrator 时显式传入 `currentActiveView`。
- store 不再隐式假设“刷新后的选中回退总是来自正常笔记列表”。

### 2.3 Node 契约测试

`sdkwork-canvas-pc-react/scripts/workspace-orchestrator.contract.test.mjs`

- 新增“当前位于 `trash` 视图刷新时，选中回退必须从回收站列表源解析”的契约测试。
- 已完成红 -> 绿验证，确保这不是事后补测。

## 3. 结果判断

本轮之后，以下能力已经被真实证明：

- `list source` 不再只由 store 的默认行为隐式决定。
- `trash` 视图的刷新选中回退已进入 orchestrator 契约。
- 工作区“快照排序 -> 列表源选择 -> activeNoteId 回退 -> 详情加载/回收站摘要回退”链路进一步收口。

## 4. 仍未完成的缺口

以下问题仍阻止 Step 04 升到 `L4`：

1. `selectedFolderId` 相关的刷新回退仍有一部分决策保留在 store。
2. 页面层仍保留快捷键与 action dispatch 编排细节。
3. repository 还没有形成面向 `read-through / replica / sync` 的策略级接口面。
