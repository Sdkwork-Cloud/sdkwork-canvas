# 06. 工作区数据源能力契约补充

- 日期：`2026-04-07`
- 适用范围：`sdkwork-canvas-pc-react / @sdkwork/canvas-canvas`
- 关联 Step：`Step 04-工作区边界收敛与数据访问抽象`

## 1. 补充目的

现有工作区已经完成基础的远端 SDK 接入与 UI 交互，但在架构层面仍需要一份明确的“数据源能力契约”，用于回答以下问题：

- 当前工作区的数据主权来自哪里
- 当前读取策略是什么
- 当前写入策略是什么
- 是否已经支持本地副本、增量同步、离线读写、冲突处理
- 未来 Step 06/07/08 扩展时应该挂接到哪一层

本补充文档用于把这一部分从“隐式实现事实”升级为“显式架构契约”。

## 2. 当前标准定义

当前 Notes 工作区的数据源能力基线定义如下：

```ts
{
  driver: 'app-sdk',
  authority: 'remote',
  readStrategy: 'workspace-snapshot',
  writeStrategy: 'direct-write',
  capabilities: {
    localReplica: false,
    readThroughCache: false,
    offlineRead: false,
    offlineWrite: false,
    backgroundSync: false,
    incrementalSync: false,
    conflictResolution: false,
  },
}
```

## 3. 设计解释

### 3.1 driver = app-sdk

说明当前工作区所有远端读写都通过统一的 App SDK 完成，而不是页面直连 HTTP，也不是仓储层自行构造另一套 transport。

架构意义：

- 统一鉴权、会话与错误处理入口
- 降低页面层耦合
- 便于后续替换 transport 时保持上层接口稳定

### 3.2 authority = remote

说明当前真实数据主权仍在远端服务端，不存在本地优先、离线优先或双主写入。

架构意义：

- 当前行为定义清晰，不伪造本地可写能力
- 决策上避免“看似支持离线，实际只是假缓存”的灰色设计

### 3.3 readStrategy = workspace-snapshot

说明工作区当前读取方式是远端快照拉取，而不是本地副本查询、增量补丁合并或搜索索引回放。

架构意义：

- 便于当前迭代保持简单、可靠
- 为后续把读取策略切换到 replica / index / incremental pipeline 提供单点替换位

### 3.4 writeStrategy = direct-write

说明当前写入路径仍是直接写远端，而不是写本地队列、延迟同步或冲突合并。

架构意义：

- 当前一致性模型清晰
- 错误处理以远端写入结果为准
- 后续队列化写入可以在不破坏上层语义的前提下扩展

## 4. 能力字段与评估标准

### 4.1 localReplica

- 当前值：`false`
- 含义：本地没有可作为权威读取面的结构化副本
- 达标条件：
  - 本地副本 schema 明确
  - 冷启动与增量更新路径明确
  - 读路径可切换到 replica
  - 副本一致性与恢复策略可验证

### 4.2 readThroughCache

- 当前值：`false`
- 含义：当前没有显式的读穿缓存策略
- 达标条件：
  - 命中与回源条件明确
  - 缓存失效策略明确
  - 读路径命中率与回源率可观测

### 4.3 offlineRead / offlineWrite

- 当前值：`false / false`
- 含义：当前不承诺离线可读或离线可写
- 达标条件：
  - 断网状态下的 UX 明确
  - 离线读的来源、容量与时效性明确
  - 离线写的队列、重放、失败回滚策略明确

### 4.4 backgroundSync / incrementalSync

- 当前值：`false / false`
- 含义：当前未启用后台同步或增量同步
- 达标条件：
  - 增量游标模型明确
  - 失败重试与幂等策略明确
  - 观测指标可证明同步正确性

### 4.5 conflictResolution

- 当前值：`false`
- 含义：当前不支持本地与远端并行写入后的冲突消解
- 达标条件：
  - 冲突检测模型明确
  - 合并策略明确
  - 用户可理解且可恢复

## 5. 主链路落点

当前该契约已经贯通以下层次：

1. `repository`
   - `queryWorkspaceSnapshot()` 负责产出数据与能力描述
2. `service`
   - 保持工作区快照类型一致性
3. `orchestrator`
   - 负责在初始化结果中对下游透传能力描述
4. `store`
   - 持有 `dataSource` 状态，供 UI、命令编排和未来同步模块消费

## 6. 架构收益

引入该契约后，当前工作区架构获得以下收益：

- 高内聚：数据能力描述集中在统一模型，不再散落在 page/store 猜测中
- 低耦合：上层不需要依赖 repository 内部实现细节判断能力
- 易扩展：后续 Step 06/07/08 可直接扩展契约，不需要重新定义边界
- 易维护：任何能力升级都可以通过契约和测试显式表达
- 易验证：`workspace-data-source.contract.test.mjs` 已成为门禁

## 7. 对 Step 06/07/08 的接口要求

### 7.1 Step 06 本地副本

进入 Step 06 时，至少需要把以下字段演进为显式支持状态：

- `localReplica`
- `readThroughCache`
- `offlineRead`

### 7.2 Step 07 搜索与索引

进入 Step 07 时，搜索模块必须基于统一能力契约判断：

- 当前搜索依赖远端快照还是本地索引
- 索引数据是否来自本地副本

### 7.3 Step 08 同步与冲突处理

进入 Step 08 时，至少需要演进以下字段：

- `backgroundSync`
- `incrementalSync`
- `conflictResolution`
- `writeStrategy`

## 8. 验收标准

本补充设计对应的实现验收标准如下：

- `NoteWorkspaceSnapshot` 必须显式包含 `dataSource`
- repository、service、orchestrator、store 必须使用同一份数据源描述语义
- `test:workspace:contracts` 必须包含工作区数据源契约测试
- TypeScript 编译必须通过
- 当前能力描述不得与真实实现不一致

## 9. 当前结论

截至 `2026-04-07`：

- 工作区数据源能力契约已经落地
- 当前工作区仍是“远端主权 + 快照读取 + 直接写入”的 `L3` 阶段实现
- Step 04 尚未完成 L4 收口，仍需继续下沉 store/page 剩余责任
