# 96. Step 并行执行周计划与排班建议

## 1. 文档定位

本文件用于把 `94` 的并行执行编排，进一步落到“周节奏、排班、检查点、收口动作”层面，便于多团队或多子 Agent 按固定节奏推进。

## 2. 适用前提

本排班建议基于以下假设：

- 主工程仍为 `sdkwork-canvas-pc-react`
- 当前阶段以重构、升级、治理为主，不是横向大功能扩张
- 能够安排统一集成 Owner
- 每周至少能完成一次集成和一次审计

## 3. 角色建议

最少角色配置：

| 角色 | 主要职责 |
| --- | --- |
| 集成 Owner | 统一边界拍板、处理冲突、收口验证 |
| 骨架/平台 Owner | 壳层、配置、安全、脚本、workflow |
| 工作区 Owner | 初始化、编辑器、本地存储、版本历史 |
| 数据能力 Owner | 搜索、同步、索引、状态机 |
| 质量 Owner | 测试、性能基准、审计、回归 |

扩展角色：

- 桌面发布 Owner
- 文档与审计 Owner

## 4. 周节奏总原则

推荐周节奏：

- 周一：冻结本周目标、输入、车道分配
- 周二到周三：并行执行
- 周四：集成、补测试、修冲突
- 周五：审计、回写、形成下一周输入

禁止做法：

- 周内没有任何中期检查点
- 多车道直到周五才首次集成
- 验证和文档回写全部压到最后一天

## 7. 多子 Agent 排班建议

### 7.1 推荐并发数

- `Wave-A`：最多 `3-4` 个并行子 Agent
- `Wave-B`：最多 `4` 个并行子 Agent
- `Wave-C`：最多 `4-5` 个并行子 Agent
- `Wave-D`：最多 `4` 个并行子 Agent

不建议无边界地超过上述并发数，因为集成和验证成本会迅速反噬效率。

### 7.2 子 Agent 分工模式

- 一个子 Agent 对应一个车道
- 一个车道只拥有一组主写入目录
- 主 Agent 只做拍板、评审、集成与收口

### 7.3 每周中期集成

如果使用多子 Agent，建议周三必须做一次中期集成检查，至少确认：

- 写入范围是否越界
- 测试与验证是否滞后
- 是否出现共享边界冲突

### 7.4 最速执行模式推荐

为了在当前仓库中尽快完成全部 Step，建议优先使用以下三档模式：

| 模式 | 适用规模 | 推荐车道组合 | 说明 |
| --- | --- | --- | --- |
| `Fast-S` | `2` 子 Agent | `主脊柱/集成` + `功能实现` | 只做有限并行，适合高稳定性优先 |
| `Fast-M` | `4` 子 Agent | `集成/文档` + `壳层/安全` + `工作区/本地层` + `搜索/同步/测试` | 当前仓库最推荐模式 |
| `Fast-L` | `6` 子 Agent | `集成` + `壳层/安全` + `初始化/编辑/本地层` + `搜索/同步` + `桌面/发布链` + `测试/性能/历史` | 速度最快，但必须严格中期集成 |

`Fast-M` 的推荐原因：

- 共享边界冲突最少
- 能同时覆盖主业务链、平台链和质量链
- 集成成本明显低于 `Fast-L`

## 8. 周度退出标准

每周结束时，至少应形成：

- 已通过的检查点列表
- 未通过的检查点及阻塞升级记录
- 已沉淀的 `docs/review/` 审计记录
- 下一周冻结输入

## 9. 结论

好的排班不是把所有人塞进同一个大目标，而是让每周目标足够清晰、车道足够独立、检查点足够前置、周五一定能形成可审计结果。本文件用于把这个节奏固化下来。
